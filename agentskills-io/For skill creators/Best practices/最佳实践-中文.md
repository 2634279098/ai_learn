# 技能创建者最佳实践

> 如何编写范围清晰且与任务相匹配的技能。

---

## 从真实专业知识出发

技能创建中常见的陷阱是：在没有提供领域特定上下文的情况下，仅依赖 LLM 的通用训练知识来生成技能。结果是模糊、通用的流程（"适当处理错误"、"遵循最佳认证实践"），而不是使技能有价值的具体 API 模式、边缘情况和项目规范。

有效的技能植根于真实的专业知识。关键是在创建过程中输入领域特定的上下文。

### 从实际任务中提取

在与 Agent 的对话中完成一个实际任务，沿途提供上下文、修正和偏好。然后将可复用的模式提取为技能。注意：

* **有效的步骤** —— 导致成功的行动序列
* **你做的修正** —— 你引导 Agent 方法的地方（例如，"使用库 X 而不是 Y"、"检查边缘情况 Z"）
* **输入/输出格式** —— 数据的输入和输出形态
* **你提供的上下文** —— Agent 原本不知道的项目特定事实、规范或约束

### 从现有项目产物中综合

当你有一堆现有知识时，可以将其输入 LLM 并要求它综合出技能。一个从你的团队实际事件报告和运维手册中综合出来的数据管道技能，会比从通用"数据工程最佳实践"文章中综合出来的技能表现更好，因为它捕获了*你的*模式、故障模式和恢复流程。关键是项目特定材料，而不是通用参考。

好的来源材料包括：

* 内部文档、运维手册和风格指南
* API 规范、模式和配置文件
* 代码审查评论和问题追踪器（捕获反复出现的问题和审查者的期望）
* 版本控制历史，特别是补丁和修复（揭示实际变更的模式）
* 真实世界的故障案例及其解决方案

## 通过实际执行来完善

技能的第一稿通常需要完善。让技能在实际任务上运行，然后将结果——包括成功和失败——反馈到创建过程中。问自己：什么触发了误报？什么被遗漏了？什么可以删减？

即使只进行一次"执行-修正"，质量也会有明显提升，复杂领域通常需要多次迭代。

> **提示**
> 阅读 Agent 的执行轨迹，而不仅仅是最终输出。如果 Agent 在不具生产力的步骤上浪费时间，常见原因包括：指令过于模糊（Agent 在找到有效方法之前尝试了多种方法）、指令不适用于当前任务（Agent 仍遵循它们），或呈现了太多选项却没有明确的默认项。

对于更有结构的迭代方法，包括测试用例、断言和评分，参见 [评估技能输出质量](/skill-creation/evaluating-skills)。

## 明智地使用上下文

一旦技能激活，其完整的 `SKILL.md` 正文将加载到 Agent 的上下文窗口中，与对话历史、系统上下文和其他激活的技能并存。技能中的每个 token 都在与窗口中其他所有内容争夺 Agent 的注意力。

### 添加 Agent 缺乏的内容，省略它已知的

专注于 Agent *没有你的技能就不会知道* 的内容：项目特定规范、领域特定流程、非显而易见的边缘情况，以及要使用的特定工具或 API。你不需要解释什么是 PDF、HTTP 如何工作、或数据库迁移是什么。

```markdown
<!-- 太啰嗦 —— Agent 已经知道 PDF 是什么 -->
## 提取 PDF 文本

PDF（可移植文档格式）文件是一种包含文本、图像和其他内容的常见文件格式。要从 PDF 中提取文本，你需要使用一个库。推荐使用 pdfplumber，因为它在大多数情况下表现良好。

<!-- 更好 —— 直接跳到 Agent 自己不会知道的内容 -->
## 提取 PDF 文本

使用 pdfplumber 进行文本提取。对于扫描文档，回退到 pdf2image 配合 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
```

对每一段内容问自己："没有这条指令，Agent 会做错吗？"如果答案是否定的，就删掉。如果不确定，就测试。如果 Agent 在没有技能的情况下已经能很好地处理整个任务，那技能可能没有增加价值。参见 [评估技能输出质量](/skill-creation/evaluating-skills) 了解如何系统性地测试。

### 设计连贯的单元

决定技能应该覆盖什么内容，就像决定函数应该做什么一样：你希望它封装一个与其他技能能良好组合的连贯工作单元。范围过窄的技能会迫使单个任务加载多个技能，增加开销和指令冲突的风险。范围过宽的技能则难以精确激活。一个用于查询数据库并格式化结果的技能可能是一个连贯单元，而一个还涵盖数据库管理的技能可能试图做得太多。

### 追求适度的详细程度

过于全面的技能可能弊大于利 —— Agent 难以提取相关内容，可能会追求由不适用于当前任务的指令触发的无效路径。简洁的、逐步指导配上工作示例，往往比详尽的文档更有效。当你发现自己涵盖了每个边缘情况时，考虑是否大部分情况由 Agent 自己的判断来处理更好。

### 用渐进式披露来构建大型技能

[规范](/specification#progressive-disclosure) 建议将 `SKILL.md` 保持在 500 行和 5,000 token 以下 —— 只需每次运行所需的核心指令。当技能确实需要更多内容时，将详细参考材料移到 `references/` 或类似目录中的单独文件。

关键是告诉 Agent *何时* 加载每个文件。"如果 API 返回非 200 状态码，请阅读 `references/api-errors.md`"比"详情请见 references/"更有用。这让 Agent 按需加载上下文，而不是预先加载，这正是 [渐进式披露](/specification#progressive-disclosure) 的设计方式。

## 校准控制程度

技能的每个部分不需要相同的规范程度。将你的指令具体性与任务的脆弱性相匹配。

### 具体性匹配脆弱性

**给 Agent 自由** 当多种方法都有效且任务可以容忍变化时。对于灵活的指令，解释*为什么*比僵硬的指令更有效 —— 理解指令背后目的的 Agent 能做出更好的情境相关决策。代码审查技能可以描述要查找什么，而不必规定具体步骤：

```markdown
## 代码审查流程

1. 检查所有数据库查询是否存在 SQL 注入（使用参数化查询）
2. 验证每个端点都有认证检查
3. 查找并发代码路径中的竞态条件
4. 确认错误消息不会泄露内部细节
```

**要规定明确** 当操作很脆弱、一致性很重要、或必须遵循特定顺序时：

```markdown
## 数据库迁移

按此顺序精确运行：

```bash
python scripts/migrate.py --verify --backup
```

不要修改命令或添加额外参数。
```

大多数技能是混合的。独立校准每个部分。

### 提供默认值，而不是菜单

当多种工具或方法都可行时，选择一个默认值，简要提一下替代方案，而不是将它们作为平等选项呈现。

```markdown
<!-- 选项太多 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...

<!-- 明确的默认方案加逃生通道 -->
使用 pdfplumber 进行文本提取：

```python
import pdfplumber
```

对于需要 OCR 的扫描 PDF，改用 pdf2image 配合 pytesseract。
```

### 偏好流程而非声明

技能应该教 Agent *如何接近* 一类问题，而不是 *为特定实例产出什么*。比较：

```markdown
<!-- 特定答案 —— 只对这个任务有用 -->
将 `orders` 表在 `customer_id` 上与 `customers` 表连接，筛选 `region = 'EMEA'`，并对 `amount` 列求和。

<!-- 可复用的方法 —— 适用于任何分析查询 -->
1. 从 `references/schema.yaml` 读取模式，找到相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何筛选条件作为 WHERE 子句应用
4. 按需聚合数值列并格式化为 markdown 表格
```

这并不意味着技能不能包含具体细节 —— 输出格式模板（参见 [输出格式模板](#输出格式模板)）、约束如"从不输出 PII"、工具特定指令都是有价值的。关键是*方法*应该通用化，即使个别细节是具体的。

## 有效指令的模式

这些是可复用的技能内容结构技术。不是每个技能都需要所有技术 —— 使用适合你任务的那些。

### 陷阱（Gotchas）章节

许多技能中最有价值的内容是一个陷阱列表 —— 违背合理假设的环境特定事实。这些不是一般建议（"适当处理错误"），而是 Agent 不被告知就会犯错的具体修正：

```markdown
## 陷阱

- `users` 表使用软删除。查询必须包含 `WHERE deleted_at IS NULL`，否则结果会包含已停用的账户。
- 用户 ID 在数据库中是 `user_id`，在认证服务中是 `uid`，在计费 API 中是 `accountId`。三者指同一个值。
- `/health` 端点只要 Web 服务器在运行就返回 200，即使数据库连接已断开。使用 `/ready` 检查完整服务健康状态。
```

将陷阱保留在 `SKILL.md` 中，让 Agent 在遇到情况前就读到。单独的参考文件也可以，如果你告诉 Agent 何时加载它，但对于非显而易见的问题，Agent 可能无法识别触发条件。

> **提示**
> 当 Agent 犯错而你需要纠正时，将纠正添加到陷阱章节。这是迭代改进技能最直接的方式之一（参见 [通过实际执行来完善](#通过实际执行来完善)）。

### 输出格式模板

当你需要 Agent 以特定格式产出时，提供模板。这比用散文描述格式更可靠，因为 Agent 对具体结构的模式匹配能力很强。短模板可以直接放在 `SKILL.md` 中；对于较长的模板，或只在特定情况下需要的模板，存放在 `assets/` 中并从 `SKILL.md` 引用，这样它们只在需要时加载。

```markdown
## 报告结构

使用此模板，根据具体分析需要调整章节：

```markdown
# [分析标题]

## 执行摘要
[关键发现的一段概述]

## 关键发现
- 带支持数据的发现 1
- 带支持数据的发现 2

## 建议
1. 具体可操作的建议
2. 具体可操作的建议
```
```

### 多步骤工作流的检查清单

明确的检查清单帮助 Agent 跟踪进度并避免跳过步骤，尤其是当步骤有依赖关系或验证关卡时。

```markdown
## 表单处理工作流

进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）
```

### 验证循环

指示 Agent 在继续前验证自己的工作。模式是：做工作、运行验证器（脚本、参考检查清单或自检）、修复问题、重复直到验证通过。

```markdown
## 编辑工作流

1. 做你的编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误消息
   - 修复问题
   - 再次运行验证
4. 只有在验证通过后才继续
```

参考文档也可以作为"验证器" —— 指示 Agent 在最终确定前对照参考检查自己的工作。

### 计划-验证-执行

对于批量或破坏性操作，让 Agent 以结构化格式创建中间计划，对照真实来源验证，然后才执行。

```markdown
## PDF 表单填写

1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段名、类型和是否必填）
2. 创建 `field_values.json`，将每个字段名映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名是否存在于表单中、类型是否兼容、必填字段是否未遗漏）
4. 如果验证失败，修正 `field_values.json` 并重新验证
5. 填写表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

关键成分是第 3 步：一个验证脚本，将计划（`field_values.json`）对照真实来源（`form_fields.json`）进行检查。像"字段 'signature_date' 未找到 —— 可用字段：customer_name、order_total、signature_date_signed"这样的错误，给 Agent 提供了足够的信息来自我纠正。

### 捆绑可复用脚本

当 [迭代改进技能](/skill-creation/evaluating-skills) 时，比较 Agent 在测试用例间的执行轨迹。如果你注意到 Agent 每次运行都独立地重新发明相同的逻辑 —— 构建图表、解析特定格式、验证输出 —— 这就是信号：编写一个经过测试的脚本，将其捆绑在 `scripts/` 中。

更多关于设计和捆绑脚本的信息，参见 [在技能中使用脚本](/skill-creation/using-scripts)。
