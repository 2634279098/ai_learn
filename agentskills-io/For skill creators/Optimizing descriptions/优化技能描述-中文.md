# 优化技能描述

> 如何改进技能的 `description` 字段，使其在相关提示下可靠触发。

---

## 技能触发机制如何工作

Agent 使用[渐进式披露](/specification#progressive-disclosure)来管理上下文。启动时，它们只加载每个可用技能的 `name` 和 `description` —— 仅足以决定技能何时可能相关。当用户任务匹配描述时，Agent 会将完整的 `SKILL.md` 读入上下文并遵循其指令。

这意味着**描述承担了触发的全部责任**。如果描述没有传达技能何时有用，Agent 就不会知道去使用它。

一个重要细节：Agent 通常只会在需要超越其自身基础能力的知识或技能时才查询技能。一个简单的一步请求如"读取这个 PDF"可能不会触发 PDF 技能，即使描述完全匹配，因为 Agent 可以用基础工具处理它。需要专业知识的任务 —— 不熟悉的 API、领域特定的工作流程或不常见的格式 —— 才是良好描述能够发挥作用的地方。

---

## 编写有效描述

在测试之前，了解好描述的样子会有所帮助。几个原则：

- **使用祈使语气。** 将描述框定为对 Agent 的指令："Use this skill when..." 而不是 "This skill does..." Agent 正在决定是否行动，所以告诉它何时行动。
- **聚焦用户意图，而非实现。** 描述用户试图实现什么，而不是技能的内部机制。Agent 根据用户要求的内容进行匹配。
- **宁可过于主动。** 明确列出技能适用的上下文，包括用户未直接命名领域的情况："even if they don't explicitly mention 'CSV' or 'analysis.'"
- **保持简洁。** 几句话到短段落通常是合适的 —— 足够覆盖技能的范围，又不会在多个技能中膨胀 Agent 的上下文。[规范](/specification#description-field) 强制规定 1024 字符的硬性限制。

## 设计触发评估查询

要测试触发，你需要一组评估查询 —— 标记了是否应该触发技能的真实用户提示。

```json
[
  { "query": "I've got a spreadsheet in ~/data/q4_results.xlsx with revenue in col C and expenses in col D — can you add a profit margin column and highlight anything under 10%?", "should_trigger": true },
  { "query": "whats the quickest way to convert this json file to yaml", "should_trigger": false }
]
```

目标约 20 个查询：8-10 个应该触发，8-10 个不应该触发。

### 应该触发的查询（Should-trigger queries）

这些测试描述是否捕获了技能的范围。沿多个维度变化：

- **措辞**：有些正式，有些随意，有些带拼写错误或缩写。
- **明确性**：有些直接命名技能领域（"analyze this CSV"），有些描述需求而不命名（"my boss wants a chart from this data file"）。
- **详细程度**：混合简短提示和上下文丰富的提示 —— 短的 "analyze my sales CSV and make a chart" 与包含文件路径、列名和背景故事的长消息并列。
- **复杂性**：变化步骤和决策点的数量。包括单步任务和多步工作流，以测试 Agent 能否在任务埋藏在更大链条中时辨别技能的相关性。

最有用的 should-trigger 查询是那些技能有帮助但关联从查询本身并不明显的情况。这些是描述措辞能够发挥作用的情况 —— 如果查询已经要求了技能所做的确切内容，任何合理的描述都会触发。

### 不应该触发的查询（Should-not-trigger queries）

最有价值的负面测试用例是**近似失误（near-misses）** —— 与技能共享关键词或概念但实际上需要不同东西的查询。这些测试描述是否精确，而不仅仅是宽泛。

对于 CSV 分析技能，弱的负面示例：

- `"Write a fibonacci function"` —— 明显无关，什么也测试不了。
- `"What's the weather today?"` —— 无关键词重叠，太简单。

强的负面示例：

- `"I need to update the formulas in my Excel budget spreadsheet"` —— 共享 "spreadsheet" 和 "data" 概念，但需要 Excel 编辑，而非 CSV 分析。
- `"can you write a python script that reads a csv and uploads each row to our postgres database"` —— 涉及 CSV，但任务是数据库 ETL，而非分析。

### 真实性建议

真实用户提示包含通用测试查询缺乏的上下文。包括：

- 文件路径（`~/Downloads/report_final_v2.xlsx`）
- 个人上下文（`"my manager asked me to..."`）
- 具体细节（列名、公司名、数据值）
- 随意语言、缩写和偶尔拼写错误

## 测试描述是否触发

基本方法：让安装了技能的 Agent 运行每个查询，观察 Agent 是否调用它。确保技能已注册且可被 Agent 发现 —— 这因客户端而异（例如，技能目录、配置文件或 CLI 标志）。

大多数 Agent 客户端提供某种形式的可观察性 —— 执行日志、工具调用历史或详细输出 —— 让你看到运行期间咨询了哪些技能。查看客户端文档了解详情。如果 Agent 加载了你的技能的 `SKILL.md`，则技能触发了；如果 Agent 未咨询就继续工作，则未触发。

查询"通过"的条件：

- `should_trigger` 为 `true` 且技能被调用，或
- `should_trigger` 为 `false` 且技能未被调用。

### 多次运行

模型行为是非确定性的 —— 同一查询可能一次触发技能，下次不触发。多次运行每个查询（3 次是合理的起点）并计算**触发率**：技能被调用的运行次数占比。

如果触发率高于阈值（0.5 是合理的默认值），should-trigger 查询通过。如果触发率低于该阈值，should-not-trigger 查询通过。

20 个查询各运行 3 次，共 60 次调用。你会想要脚本化。以下是一般结构 —— 将 `claude` 调用和 `check_triggered` 中的检测逻辑替换为你的 Agent 客户端提供的内容：

```bash
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# 此示例使用 Claude Code 的 JSON 输出检查 Skill 工具调用。
# 将此函数替换为你的 Agent 客户端的检测逻辑。
# 如果技能被调用应返回 0（成功），否则返回 1。
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

> **提示：** 如果你的 Agent 客户端支持，可以在结果明确后提前停止运行 —— Agent 要么咨询了技能，要么未咨询就开始工作。这可以显著减少运行完整评估集的时间和成本。

## 用训练/验证集划分避免过拟合

如果针对所有查询优化描述，你有过拟合的风险 —— 设计出仅对这些特定措辞有效但在新措辞上失败的描述。

解决方案是划分查询集：

- **训练集（~60%）**：用于识别失败并指导改进的查询。
- **验证集（~40%）**：留出的查询，仅用于检查改进是否泛化。

确保两个集都包含成比例的 should-trigger 和 should-not-trigger 查询 —— 不要意外将所有正面查询放入一个集合。随机打乱并保持划分固定，以便跨迭代比较。

如果你使用[上文](#多次运行)中的脚本，可以将查询分成两个文件 —— `train_queries.json` 和 `validation_queries.json` —— 分别对两个文件运行脚本。

## 优化循环

1. **在训练集和验证集上评估**当前描述。训练结果指导你的更改；验证结果告诉你这些更改是否在泛化。
2. 在**训练集**上**识别失败**：哪些 should-trigger 查询没有触发？哪些 should-not-trigger 查询错误触发了？
   - 仅使用训练集失败来指导你的更改 —— 无论你是自己修改描述还是提示 LLM，都让验证集结果不参与此过程。
3. **修改描述。** 专注于泛化：
   - 如果 should-trigger 查询失败，描述可能太窄。扩大范围或添加关于技能何时有用的上下文。
   - 如果 should-not-trigger 查询错误触发，描述可能太宽。添加关于技能*不做什么*的具体性，或澄清此技能与相邻能力之间的边界。
   - 避免添加失败查询中的特定关键词 —— 那是过拟合。相反，找到这些查询代表的一般类别或概念并解决它。
   - 如果几次迭代后卡住，尝试结构上不同的描述方法，而不是增量调整。不同的框架或句子结构可能在精细化无法突破的地方取得进展。
   - 检查描述是否保持在 1024 字符限制内 —— 描述在优化过程中往往会增长。
4. **重复**步骤 1-3，直到所有**训练集**查询通过或停止看到有意义的改进。
5. **选择验证通过率最高**的迭代 —— 即**验证集**中通过的查询比例。注意最佳描述可能不是你最后生成的；较早的迭代可能比较晚过拟合训练集的迭代具有更高的验证通过率。

五次迭代通常足够了。如果性能没有提高，问题可能出在查询（太简单、太难或标记错误）而非描述本身。

> **提示：** [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 技能可端到端自动化此循环：分割评估集、并行评估触发率、使用 Claude 提出描述改进建议，并生成实时 HTML 报告供你观看。

## 应用结果

一旦选择了最佳描述：

1. 更新 `SKILL.md` frontmatter 中的 `description` 字段。
2. 验证描述是否在[1024 字符限制](/specification#description-field)内。
3. 验证描述是否按预期触发。手动尝试几个提示作为快速检查。对于更严格的测试，编写 5-10 个新查询（混合 should-trigger 和 should-not-trigger）并通过评估脚本运行它们 —— 由于这些查询从未参与优化过程，它们能诚实地检查描述是否泛化。

**前后对比：**

```yaml
# Before
description: Process CSV files.

# After
description: >
  Analyze CSV and tabular data files — compute summary statistics,
  add derived columns, generate charts, and clean messy data. Use this
  skill when the user has a CSV, TSV, or Excel file and wants to
  explore, transform, or visualize the data, even if they don't
  explicitly mention "CSV" or "analysis."
```

改进后的描述对技能的功能更具体（摘要统计、派生列、图表、清洗），对适用场景更宽泛（CSV、TSV、Excel；即使没有明确关键词）。
