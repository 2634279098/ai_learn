# AI 编程心得



## 精准提问：让 AI 输出「能用的代码」而非「半成品」

1. **结构化指令，拒绝模糊描述**

   - ❌ 错误示例：“写一个 Python 接口”

   - ✅ 正确示例：`用FastAPI写一个GET接口，路径/api/user/{id}，返回JSON格式（包含id/name/phone），要求：① 参数校验 ② 异常捕获 ③ 注释完整 ④ 附带调用示例`

   - 核心：明确**语言 / 框架 + 功能 + 约束（性能 / 兼容 / 规范） + 输出格式**。

   

2. **限定代码风格与规范**

   - 明确要求：“代码符合 PEP8 规范（Python）/ 阿里巴巴 Java 开发手册，变量名用驼峰命名，函数需加 docstring 注释”。

   - 适配场景：“代码需兼容 Python 3.8+，不使用第三方库（除 requests）”。

   

3. **分阶段拆解复杂需求**
   - 复杂项目（如电商订单系统）：先让 AI 输出**架构思路** → 再拆分为「数据库设计」「接口层」「业务逻辑层」→ 最后逐模块生成代码，避免一次性输出冗余内容。



## 高效交互：最大化 AI 编程能力

1. **利用「长上下文」优势（Kimi/DeepSeek/Claude）**

   - 直接粘贴**完整代码文件 / 报错日志 / 需求文档**，让 AI 基于上下文修改，而非只描述问题（例：“修复以下代码中的空指针异常，并解释原因：[粘贴代码]”）。

   - 对大型代码库：让 AI 先 “总结代码结构” → 再 “修改指定模块”，减少理解成本。

   

2. **善用「思考链」提示词**

   - 给 AI 加前置指令：“先分析需求的核心逻辑，列出实现步骤，再写代码，最后说明关键逻辑”。
     - 简单说，**思考链就是让 AI 先 “想明白” 再 “写代码”**，避免它直接给你一段没逻辑的半成品。

   - 例：`请实现一个Redis分布式锁，要求：① 防死锁 ② 可重入 ③ 自动续期。步骤：1. 分析分布式锁的核心问题 2. 设计实现思路 3. 编写Python代码 4. 说明注意事项`。

   

3. **主动要求「多方案对比」**
   - 对关键模块（如排序 / 缓存）：“用两种方式实现（原生代码 / 第三方库），对比性能和适用场景”，避免 AI 默认选择 “最简单但不最优” 的方案。



## 避坑与验证：AI 代码≠直接上线

1. **必做的人工校验点**

   - ✅ 语法 / 逻辑错误：AI 易漏边界条件（如空值、异常、并发），需重点检查循环 / 判断 / IO 操作。
   - ✅ 安全问题：AI 生成的接口易忽略鉴权 / 参数过滤，需补充 “XSS/CSRF 防护”“SQL 注入校验”。
   - ✅ 性能：AI 默认写 “能用的代码”，需优化循环次数、数据结构（例：列表遍历→字典查找）。

   

2. **用 AI 自查 AI 代码**

   - 让 AI 输出代码后，追加指令：“检查这段代码的潜在问题（性能 / 安全 / 兼容性），并给出优化方案”。
   - 例：`分析以下代码的性能瓶颈，优化后给出新版本，并说明优化点：[粘贴AI生成的代码]`。

   

3. **小步迭代，逐步验证**

   - 先让 AI 生成「最小可运行示例（MVP）」→ 本地跑通 → 再让 AI 扩展功能，避免一次性生成大量代码难以调试。



## 如何保证 AI coding 符合项目规范

**1. 提需求时：明确规范 + 给参考**

- *写清规则：**在提示词里直接定义命名、缩进、注释等要求，比如：

> 写 Python 接口，变量用蛇形（user_id），4 空格缩进，日志用项目 logging，函数加 docstring。

- **贴项目代码**：复制一段项目里的规范代码当模板，让 AI 对齐风格。



**2. 生成后：自动化校准 + 人工核对**

- **工具自动修正**：用项目格式化工具（如 Black、Prettier）改缩进 / 换行，用 Lint 工具（如 flake8）查命名 / 违规写法。
- **人工重点检查**：确认命名风格、代码结构、公共工具复用是否和项目一致。



3. **长期：固化模板**

- 整理项目专属提示词模板，每次提需求直接套用；
- 把修正后的代码反馈给 AI，让它记住项目规范。
