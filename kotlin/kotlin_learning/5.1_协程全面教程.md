## 1. 协程是什么？为什么需要它？

### 1.1 遇到的问题

在传统的线程模型中，处理异步操作（如网络请求、数据库读写）有很多痛点：

```kotlin
// 传统的回调方式——"回调地狱"
fetchUser { user ->
    fetchUserProfile(user.id) { profile ->
        fetchFriends(profile.id) { friends ->
            // 嵌套越来越深，代码难以维护
        }
    }
}
```

**线程的问题：**
- 创建线程开销大（每个线程约 1MB 栈内存）
- 线程切换成本高（上下文切换）
- 线程数量有限，容易阻塞

### 1.2 协程的答案

**协程（Coroutine）= 轻量级的线程**

| 对比 | 线程（Thread） | 协程（Coroutine） |
|------|---------------|-------------------|
| 开销 | ~1MB 每线程 | ~几十字节每协程 |
| 并发量 | 几千个就很多了 | 几十万个没问题 |
| 切换成本 | 操作系统级切换，成本高 | 用户态切换，成本极低 |
| 语法 | 回调/Future 嵌套 | 看似同步的写法 |

**核心优势：用同步的写法，实现异步的效果**

```kotlin
// 协程写法——看似同步，实际异步
suspend fun loadUserData() {
    val user = fetchUser()      // 挂起，但不阻塞线程
    val profile = fetchProfile(user.id)
    val friends = fetchFriends(profile.id)
    // 代码是顺序写的，但底层是非阻塞的！
}
```

### 1.3 关键概念预览

- **挂起（Suspend）**：协程可以在不阻塞线程的情况下暂停执行
- **恢复（Resume）**：挂起后可以稍后在合适的线程上继续执行
- **非阻塞**：挂起时，底层线程可以去执行其他协程

## 2. 第一个协程程序

### 2.1 添加依赖

```gradle
// build.gradle.kts (Android)
dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-coroutines-android:1.7.3")
}

// 或纯 Kotlin 项目
dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.7.3")
}
```

### 2.2 Hello Coroutine

```kotlin
import kotlinx.coroutines.*

fun main() = runBlocking {
    // runBlocking：创建一个协程作用域，并阻塞当前线程直到内部所有协程完成
    
    launch {
        // launch：在后台启动一个新的协程
        delay(1000L)  // delay：非阻塞的等待（挂起函数）
        println("World!")
    }
    
    println("Hello,")
    // 这里会等待 launch 中的代码执行完毕（因为 runBlocking 会等待）
}

// 输出：
// Hello,
// （等待约1秒）
// World!
```

**关键点：**
- `runBlocking`：桥接普通代码和协程世界的"入口"
- `launch`：启动一个不需要返回结果的协程
- `delay(timeMillis)`：非阻塞地等待指定时间（挂起函数）

### 2.3 结构化并发（Structured Concurrency）

协程不是"野线程"——它们有明确的**作用域**：

```kotlin
runBlocking {              // 外层作用域
    launch {               // 子协程1
        delay(1000L)
        println("Child 1 done")
    }
    launch {               // 子协程2
        delay(500L)
        println("Child 2 done")
    }
    println("Waiting...")
}
// 输出：
// Waiting...
// Child 2 done  （500ms后）
// Child 1 done  （1000ms后）
// runBlocking 在所有子协程完成后才结束
```

**结构化并发的好处：**
- 父协程会等待所有子协程完成
- 父协程取消时，所有子协程也会被取消
- 子协程出错时，父协程和其他子协程也会被取消

## 3. 挂起函数（suspend）

### 3.1 suspend 关键字

`suspend` 是协程的核心——它标记一个函数"可以挂起"。

```kotlin
// 挂起函数：可以在不阻塞线程的情况下暂停执行
suspend fun fetchUserData(): User {
    delay(1000L)  // 模拟网络请求
    return User("Alice", 25)
}

// 挂起函数只能在协程或其他挂起函数中调用
fun main() = runBlocking {
    val user = fetchUserData()  // ✅ 在协程中调用
    println("User: ${user.name}")
}

// ❌ 错误：不能在普通函数中直接调用挂起函数
fun normalFunction() {
    // fetchUserData()  // 编译错误！
}
```

### 3.2 挂起的原理（通俗理解）

```text
普通函数：  start ────────────────── end  （一直占用线程）

挂起函数：  start ──挂起──> 释放线程
                     │
                     │  （线程可以去执行其他任务）
                     │
                     └── 恢复 ──> 继续在合适线程执行 ── end
```

**类比：**
- 普通函数 = 你在窗口排队，必须等到办完才能走
- 挂起函数 = 你取了个号，去旁边坐着等，窗口可以服务其他人

### 3.3 常见的挂起函数

```kotlin
suspend fun examples() = coroutineScope {
    delay(1000L)           // 挂起等待指定时间
    yield()                 // 主动让出执行权，让其他协程运行
    withContext(Dispatchers.IO) {  // 切换到 IO 线程执行
        // ...
    }
    val result = withTimeout(5000L) {  // 设置超时
        // ...
    }
}
```

### 3.4 挂起函数 ≠ 多线程

**重要：** 挂起函数本身不意味着切换线程！

```kotlin
suspend fun wrongWay() {
    // ❌ 错误理解：suspend 会自动切换到后台线程
    delay(1000L)
    // 这里仍然在调用者的线程上！
}

suspend fun rightWay() = withContext(Dispatchers.IO) {
    // ✅ 正确：显式切换到 IO 线程
    // 执行网络请求或数据库操作
}
```

## 4. 协程作用域与构建器

### 4.1 协程构建器对比

Kotlin 提供了几个主要的协程构建器：

| 构建器 | 返回值 | 用途 | 异常处理 |
|--------|--------|------|---------|
| `launch` | `Job` | 执行不需要返回值的任务 | 异常立即抛出 |
| `async` | `Deferred<T>` | 执行需要返回值的任务 | 异常在 await 时抛出 |
| `runBlocking` | 最后一行的值 | 桥接普通代码和协程（仅用于 main 或测试） | 异常立即抛出 |
| `coroutineScope` | 最后一行的值 | 在挂起函数中创建子协程作用域 | 任何子协程失败都会取消其他子协程 |

### 4.2 launch—— fire and forget

```kotlin
fun main() = runBlocking {
    val job: Job = launch {
        println("Working...")
        delay(1000L)
        println("Done!")
    }
    
    // 可以对 Job 进行操作
    // job.cancel()    // 取消协程
    // job.join()      // 等待协程完成
    // job.isActive    // 检查是否还在运行
}
```

### 4.3 async/await—— 获取返回值

```kotlin
fun main() = runBlocking {
    val deferred: Deferred<Int> = async {
        delay(1000L)
        42  // 返回值
    }
    
    println("Waiting for result...")
    val result = deferred.await()  // 等待并返回结果
    println("Result: $result")    // 输出：Result: 42
}
```

### 4.4 coroutineScope—— 结构化并发的关键

```kotlin
suspend fun loadMultipleData() = coroutineScope {
    // 在这个作用域内启动多个并发任务
    val userDeferred = async { fetchUser() }
    val postsDeferred = async { fetchPosts() }
    
    // 等待所有任务完成
    val user = userDeferred.await()
    val posts = postsDeferred.await()
    
    // 如果 fetchUser 或 fetchPosts 任何一个失败，
    // 另一个会被自动取消（结构化并发）
    
    UserWithPosts(user, posts)
}
```

### 4.5 作用域层级关系

```
runBlocking (根作用域)
├── launch (子协程1)
│   └── async (孙子协程)
├── launch (子协程2)
│   └── coroutineScope
│       └── launch (孙协程)
└── 当根作用域取消时，所有子协程和孙协程都会被取消
```

## 5. 调度器（Dispatchers）

### 5.1 什么是调度器？

**调度器（Dispatcher）决定协程运行在哪个线程（或线程池）上。**

```kotlin
fun main() = runBlocking {
    launch(Dispatchers.Default) {
        // 计算密集型任务（CPU 密集型）
        println("Default: ${Thread.currentThread().name}")
    }
    
    launch(Dispatchers.IO) {
        // IO 密集型任务（网络、数据库）
        println("IO: ${Thread.currentThread().name}")
    }
    
    launch(Dispatchers.Main) {
        // Android 主线程（UI 线程）
        println("Main: ${Thread.currentThread().name}")
    }
    
    launch(Dispatchers.Unconfined) {
        // 不限定线程（不推荐常规使用）
        println("Unconfined: ${Thread.currentThread().name}")
    }
}
```

### 5.2 四种调度器详解

#### Dispatchers.Default
```kotlin
// 适合：CPU 密集型计算
// 线程数 = CPU 核心数（最少 2 个）
launch(Dispatchers.Default) {
    val result = (1..1_000_000).count { it % 2 == 0 }
    println("Even count: $result")
}
```

#### Dispatchers.IO
```kotlin
// 适合：网络请求、数据库操作、文件读写
// 线程数默认 = 64 或 CPU 核心数的较大值
launch(Dispatchers.IO) {
    val response = api.getUser()
    val data = database.loadData()
    // ...
}
```

#### Dispatchers.Main（Android）
```kotlin
// 适合：UI 更新操作
// 只能在 Android 项目中使用
launch(Dispatchers.Main) {
    textView.text = "Loading..."
    val data = withContext(Dispatchers.IO) {
        api.fetchData()  // 在 IO 线程执行
    }
    textView.text = data  // 回到主线程更新 UI
}
```

#### Dispatchers.Unconfined
```kotlin
// 不推荐常规使用
// 协程在哪个线程恢复，就在哪个线程继续执行
// 主要用于测试或特殊场景
```

### 5.3 withContext—— 切换上下文

`withContext` 是最常用的切换线程的方式：

```kotlin
suspend fun loadUser(): User = withContext(Dispatchers.IO) {
    // 整个代码块在 IO 线程执行
    api.getUser()
}
// withContext 会挂起，直到代码块执行完毕，并返回结果
// 调用者线程在挂起期间不会被阻塞！

// 实际使用模式（Android 经典模式）：
suspend fun updateUserUI(): Unit = withContext(Dispatchers.Main) {
    val user = withContext(Dispatchers.IO) {
        api.getUser()  // IO 线程
    }
    // 自动回到 Main 线程
    textView.text = user.name  // 更新 UI
}
```

### 5.4 调度器选择决策树

```
需要执行什么类型的任务？
│
├── CPU 密集型（计算、排序、图像处理）
│   └── Dispatchers.Default
│
├── IO 密集型（网络、数据库、文件）
│   └── Dispatchers.IO
│
├── UI 更新（Android）
│   └── Dispatchers.Main
│
└── 不清楚 / 不关心
    └── 不指定，继承父协程的调度器
```

## 6. async/await 与并发

### 6.1 串行 vs 并行

```kotlin
// ❌ 串行执行（慢）
suspend fun loadDataSerially() = coroutineScope {
    val user = async { fetchUser() }.await()    // 先执行这个
    val posts = async { fetchPosts() }.await()  // 再执行这个
    // 总耗时 = fetchUser 时间 + fetchPosts 时间
}

// ✅ 并行执行（快）
suspend fun loadDataConcurrently() = coroutineScope {
    val userDeferred = async { fetchUser() }    // 启动协程1
    val postsDeferred = async { fetchPosts() }   // 启动协程2（同时执行！）
    
    val user = userDeferred.await()   // 等待协程1完成
    val posts = postsDeferred.await() // 等待协程2完成
    // 总耗时 = max(fetchUser 时间, fetchPosts 时间)
}
```

### 6.2 await() 的调用时机

```kotlin
suspend fun examples() = coroutineScope {
    // 方式1：先启动所有，最后统一 await（推荐，最大化并行）
    val d1 = async { task1() }
    val d2 = async { task2() }
    val d3 = async { task3() }
    val r1 = d1.await()
    val r2 = d2.await()
    val r3 = d3.await()
    
    // 方式2：启动后立即 await（相当于串行）
    val r1 = async { task1() }.await()  // 等待完成
    val r2 = async { task2() }.await()  // 再启动下一个
    // 这样写就失去了并行的意义！
}
```

### 6.3 处理并行中的异常

```kotlin
suspend fun loadDataWithFallback() = coroutineScope {
    val userDeferred = async { fetchUser() }
    val cachedDeferred = async { loadFromCache() }
    
    try {
        userDeferred.await()
    } catch (e: Exception) {
        // 如果网络请求失败，使用缓存
        cachedDeferred.await()
    }
}
```

### 6.4 lazy 异步

```kotlin
suspend fun lazyAsync() = coroutineScope {
    val deferred = async(start = CoroutineStart.LAZY) {
        // 这个代码块不会立即执行
        fetchData()
    }
    
    // ... 做一些其他事情 ...
    
    deferred.start()  // 手动启动
    // 或
    val result = deferred.await()  // await 也会触发启动
}
```

## 7. 协程取消与超时

### 7.1 协程取消原理

协程的取消是**协作式的**——协程代码需要"配合"才能被取消。

```kotlin
fun main() = runBlocking {
    val job = launch {
        repeat(1000) { i ->
            println("Working... $i")
            delay(500L)  // delay 是会检查取消的挂起函数
        }
    }
    
    delay(1300L)    // 主线程等待一会儿
    job.cancel()     // 取消协程
    job.join()       // 等待协程真正结束
    println("Cancelled!")
}

// 输出：
// Working... 0
// Working... 1
// Working... 2
// Cancelled!
```

### 7.2 为什么需要协作式取消？

```kotlin
// ❌ 这个协程无法被取消！
launch {
    var i = 0
    while (i < 1000) {  // 没有挂起函数调用，无法检查取消状态
        println("Working... $i")
        i++
        // 这个循环会一直跑，即使调用了 cancel()
    }
}

// ✅ 正确方式1：使用挂起函数
launch {
    var i = 0
    while (i < 1000) {
        println("Working... $i")
        delay(1L)  // delay 会检查取消状态
        i++
    }
}

// ✅ 正确方式2：检查取消状态
launch {
    var i = 0
    while (isActive) {  // isActive 是 CoroutineScope 的属性
        println("Working... $i")
        // 做一些工作
        i++
    }
}

// ✅ 正确方式3：yield()
launch {
    var i = 0
    while (i < 1000) {
        println("Working... $i")
        yield()  // 让出执行权，并检查取消
        i++
    }
}
```

### 7.3 处理取消异常

```kotlin
fun main() = runBlocking {
    val job = launch {
        try {
            repeat(1000) { i ->
                println("Working... $i")
                delay(500L)
            }
        } catch (e: CancellationException) {
            println("Cancelled! Cleaning up...")
            // 执行清理工作
            throw e  // 重要：重新抛出，让取消继续传播
        } finally {
            println("Finally block - cleanup")
            // 注意：finally 中的代码默认不能调用挂起函数
            // 如果需要，用 withContext(NonCancellable) { ... }
        }
    }
    
    delay(1300L)
    job.cancelAndJoin()  // 取消并等待
}
```

### 7.4 withTimeout—— 设置超时

```kotlin
suspend fun fetchWithTimeout() = coroutineScope {
    try {
        val result = withTimeout(5000L) {  // 5秒超时
            fetchData()  // 如果5秒内没完成，会抛出 TimeoutCancellationException
        }
        result
    } catch (e: TimeoutCancellationException) {
        // 超时了，返回默认值
        defaultData
    }
}

// 不抛异常的版本：
suspend fun fetchWithTimeoutOrNull() = coroutineScope {
    val result = withTimeoutOrNull(5000L) {
        fetchData()
    }
    result ?: defaultData  // 超时时返回 null
}
```

## 8. 异常处理

### 8.1 协程中的异常传播

```kotlin
fun main() = runBlocking {
    // launch：异常会立即抛出，导致协程崩溃
    val job = launch {
        println("Starting...")
        throw RuntimeException("Oops!")  // 这里崩溃
        println("This won't print")
    }
    
    // async：异常会在 await() 时抛出
    val deferred = async {
        throw RuntimeException("Oops in async!")
        42
    }
    
    try {
        deferred.await()  // 这里才会抛出异步中的异常
    } catch (e: RuntimeException) {
        println("Caught: ${e.message}")
    }
}
```

### 8.2 CoroutineExceptionHandler

```kotlin
fun main() = runBlocking {
    val exceptionHandler = CoroutineExceptionHandler { _, exception ->
        println("Caught exception: ${exception.message}")
    }
    
    val job = GlobalScope.launch(exceptionHandler) {
        throw RuntimeException("Oops!")  // 异常会被 handler 捕获
    }
    
    job.join()
}
```

**注意：** `CoroutineExceptionHandler` 只在以下情况生效：
- 使用 `launch` 启动的协程
- 异常没有在其他地方被处理
- 不是 `async` 的异常（async 的异常在 await 时抛出）

### 8.3 结构化并发中的异常传播

```kotlin
fun main() = runBlocking {
    // 在 coroutineScope 中，任何一个子协程失败，
    // 会导致整个 scope 取消（所有其他子协程也会被取消）
    try {
        coroutineScope {
            launch {
                delay(1000L)
                println("Child 1 done")
            }
            launch {
                delay(500L)
                throw RuntimeException("Child 2 failed!")
            }
        }
    } catch (e: RuntimeException) {
        println("Scope failed: ${e.message}")
        // Child 1 也会被取消，不会打印 "Child 1 done"
    }
}
```

### 8.4 supervisorScope—— 隔离异常

```kotlin
fun main() = runBlocking {
    // supervisorScope：子协程相互独立，一个失败不会影响其他
    supervisorScope {
        launch {
            delay(1000L)
            println("Child 1 done")  // 这个仍然会执行
        }
        launch {
            delay(500L)
            throw RuntimeException("Child 2 failed!")
        }
    }
    // 与 coroutineScope 不同，这里 child 1 不会被取消
}
```

### 8.5 异常处理最佳实践

```kotlin
// ✅ 推荐：在顶层协程处理异常
fun main() = runBlocking {
    try {
        val data = fetchData()
        updateUI(data)
    } catch (e: Exception) {
        showError(e.message)
    }
}

// ✅ 推荐：使用 CoroutineExceptionHandler 处理未捕获的异常
val handler = CoroutineExceptionHandler { _, exception ->
    Log.e("Coroutine", "Uncaught exception", exception)
}

// ❌ 避免：在每一个挂起函数内部处理异常
// （应该在调用处统一处理）
```

## 9. Flow——协程版的 RxJava

### 9.1 为什么需要 Flow？

`suspend` 函数只能返回一个值。如果需要**多次返回**（就像数据流），就需要 Flow：

```kotlin
// suspend 函数：返回单个值
suspend fun fetchUser(): User { ... }

// Flow：返回多个值（异步流）
fun fetchUsersFlow(): Flow<User> { ... }
```

### 9.2 第一个 Flow

```kotlin
import kotlinx.coroutines.flow.*

fun simpleFlow(): Flow<Int> = flow {
    for (i in 1..3) {
        delay(1000L)  // 模拟异步操作
        emit(i)       // 发射一个值
    }
}

fun main() = runBlocking {
    simpleFlow()
        .collect { value ->  // 收集流中的值
            println(value)
        }
}
// 输出（每秒输出一个）：
// 1  （1秒后）
// 2  （2秒后）
// 3  （3秒后）
```

### 9.3 Flow 的操作符

```kotlin
fun main() = runBlocking {
    (1..10).asFlow()
        .filter { it % 2 == 0 }    // 过滤：只保留偶数
        .map { it * it }            // 变换：平方
        .take(3)                    // 只取前3个
        .collect { println(it) }    // 收集
}
// 输出：4, 16, 36
```

### 9.4 Flow 的线程切换

```kotlin
fun flowWithContext(): Flow<Int> = flow {
    // flow 构建器中可以使用 withContext 切换线程
    for (i in 1..3) {
        delay(1000L)
        emit(i)
    }
}.flowOn(Dispatchers.IO)  // 指定上游操作在 IO 线程执行

// 或者使用更灵活的方式：
fun anotherFlow(): Flow<Int> = flow {
    for (i in 1..3) {
        withContext(Dispatchers.IO) {
            // 在 IO 线程执行
            emit(fetchData(i))
        }
    }
}
```

    Result.Error("HTTP error: ${e.code()}")

## 11. 最佳实践与常见陷阱

### 11.1 最佳实践

```kotlin
// ✅ 1. 使用合适的 CoroutineScope
// Android: viewModelScope, lifecycleScope
// 后端: 自定义 CoroutineScope，记得在适当时候取消

// ✅ 2. 明确指定调度器
suspend fun fetchData() = withContext(Dispatchers.IO) {
    api.getData()
}

// ✅ 3. 正确处理异常
viewModelScope.launch {
    try {
        val data = fetchData()
        _state.value = UiState.Success(data)
    } catch (e: Exception) {
        _state.value = UiState.Error(e.toUserMessage())
    }
}

// ✅ 4. 避免 GlobalScope（除非你明确需要全局协程）
// ❌ 不推荐：GlobalScope.launch { ... }
// ✅ 推荐：使用有生命周期的作用域

// ✅ 5. 使用 suspend 函数而不是 callback
// ❌ 不推荐
fun fetchUser(callback: (User) -> Unit) {
    api.getUser(object : Callback<User> {
        override fun onSuccess(user: User) { callback(user) }
        override fun onFailure(e: Exception) { ... }
    })
}
// ✅ 推荐
suspend fun fetchUser(): User = withContext(Dispatchers.IO) {
    api.getUser()
}
```

### 11.2 常见陷阱

#### 陷阱1：在协程中执行阻塞操作

```kotlin
// ❌ 错误：在协程中执行阻塞 IO
launch(Dispatchers.Main) {
    val data = api.fetchData()  // 如果是同步网络请求，会阻塞主线程！
}

// ✅ 正确：切换到 IO 线程
launch(Dispatchers.Main) {
    val data = withContext(Dispatchers.IO) {
        api.fetchData()  // 在 IO 线程执行
    }
    updateUI(data)
}
```

#### 陷阱2：忘记处理异常

```kotlin
// ❌ 危险：launch 中的异常会导致应用崩溃
launch {
    val data = api.fetchData()  // 如果这里抛异常，整个协程会崩溃
    updateUI(data)
}

// ✅ 安全：处理异常
launch {
    try {
        val data = api.fetchData()
        updateUI(data)
    } catch (e: Exception) {
        showError(e.message)
    }
}
```

#### 陷阱3：协程泄漏

```kotlin
// ❌ 错误：协程可能在 Activity 销毁后还在运行
fun loadData() {
    GlobalScope.launch {
        val data = api.fetchData()
        updateUI(data)  // Activity 可能已经销毁了！
    }
}

// ✅ 正确：使用生命周期感知的作用域
fun loadData() {
    lifecycleScope.launch {
        val data = api.fetchData()
        updateUI(data)  // 安全：Activity 销毁时协程会自动取消
    }
}
```

#### 陷阱4：过度并行

```kotlin
// ❌ 危险：同时发起 1000 个网络请求
suspend fun loadAll(ids: List<String>) = coroutineScope {
    ids.map { id ->
        async { api.fetchItem(id) }  // 同时启动 1000 个协程！
    }.awaitAll()
}

// ✅ 正确：限制并发数量
suspend fun loadAll(ids: List<String>) = coroutineScope {
    ids.chunked(10).flatMap { chunk ->
        chunk.map { id ->
            async { api.fetchItem(id) }
        }.awaitAll()
    }
}
```
