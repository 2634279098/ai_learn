## 一、Vibe Coding 概念

**定义：** 你描述意图，AI 写代码，你审查结果、给反馈、推进方向。

**关键特征：**
- 整个编码过程都在和 AI 交互，是连续的对话流，不是孤立的问答
- 覆盖完整流程：需求拆解 → 写代码 → 改 bug → 写测试

**成立条件：**
1. **AI 的代码能力够用**（生成的东西改改就能用）—— 2025 年后基本满足，Claude Opus 4.6、GPT-5.4 可用于实际工作
2. **工具链够顺**（延迟低、上下文不丢、操作不别扭）—— 这是大多数人真正需要解决的问题

**起源：** Andrej Karpathy 2025 年在推特描述自己写代码完全不关注语法，只关注想解决什么问题，全靠 AI 生成代码，自己只负责审查和方向调整。

## 二、工具生态对比

| 工具 | 定位 | 优势 | 局限 |
|------|------|------|------|
| **Claude Code** | 深度任务首选（Anthropic 终端 CLI） | 代码库理解深、多文件联动修改、能真正执行任务（读写文件、跑命令、跑测试） | 成本较高 |
| **Cursor** | 日常补全（AI 增强 VS Code） | Tab 补全流畅、迁移成本低、Cmd+K/L 交互方便 | 项目级理解不如 Claude Code，跨文件复杂任务可靠性差 |
| **OpenClaw** | 多任务并行（开源 AI 编程框架） | 多 Agent 并行执行、任务队列、自定义 skill | 上手复杂度较高 |

### 推荐组合

- **日常：** Cursor（行级补全+小改动）+ Claude Code（模块重构+复杂任务）
- **Claude Code 模型选择：** Sonnet 4.6 做日常任务，Opus 4.6 做复杂重构
- **大型重构/架构调整：** Claude Code 配 Opus 4.6（单次 $10-30）
- **多任务并行/自动化：** OpenClaw Agent 工作流

## 三、最佳实践

### 1. 任务描述要清晰

- ❌ "帮我写一个用户注册功能"
- ✅ "我在做 Node.js + PostgreSQL 的 SaaS 项目，ORM 用 Prisma，帮我实现用户注册，要有邮箱验证，错误处理跟已有 middleware 风格保持一致"

**上下文越清楚，来回改的次数越少**

### 2. 必须做 Code Review

- AI 生成初稿 → 快速 Review 关键逻辑 → 提问题要求改进 → AI 修改 → 确认
- **不能不看生成的代码直接提交**，可能有逻辑漏洞、安全问题或业务理解偏差

### 3. 积累项目上下文（CLAUDE.md）

- 在项目根目录放 `CLAUDE.md`，写上项目背景、技术决策、代码规范
- 每次启动 Claude Code 自动加载，不用重复解释背景

### 4. 保持节奏，避免上下文碎片化

- 把功能拆成小任务，一次只做一件事
- 在同一个 session 里推进同一主题，不频繁切换话题
- 多任务可用 OpenClaw 任务队列排队执行

## 四、场景指南

| 场景 | 推荐工具 | 说明 |
|------|----------|------|
| 日常业务功能开发 | Cursor + Claude Code | Cursor 节奏快，跨文件多切 Claude Code |
| 大型重构/架构调整 | Claude Code + Opus 4.6 | 需要深度理解，预估成本 $10-30 |
| 独立开发/快速原型 | Claude Code | 快速出原型可以，但核心架构要自己想清楚 |
| 团队协作/CI 集成 | OpenClaw Agent 工作流 | 接入 CI/CD，自动跑测试、处理 review 反馈 |

## 六、核心理念

> **分工原则：你想清楚要做什么，AI 帮你做。**

- AI 在**执行层面**很强，**方向判断**上不靠谱
- 功能该不该做、架构怎么设计、技术债是否值得还 —— 这些是人的工作
- 完全依赖 AI 做决策会导致系统越来越难维护

**最终建议：** 主力工具选一个学透 > 两个都学一半；配置好 API 中转；任务描述清晰；生成代码照样要 Review。

## 📌 实践要点总结

1. ✅ 上下文要详细，技术栈、约束、风格都要说清楚
2. ✅ 生成代码必须 Review，不能直接提交
3. ✅ 建立 CLAUDE.md 积累项目上下文
4. ✅ 任务拆分小步快跑，避免上下文碎片化
5. ✅ 主力工具学透，不要工具链全学但都只学一半
