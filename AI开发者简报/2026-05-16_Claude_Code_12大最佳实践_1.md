# Claude Code 12 大最佳实践

## 一句话总结
从"跟风编程"迈向"Agent 工程"——通过系统化配置 CLAUDE.md、Agent、Skills、Hooks 等内置能力，让 Claude Code 自动执行工作流并自我验证，而非每次手写提示词。

## 适用场景
- 使用 Claude Code 进行日常开发的程序员
- 希望将 AI 编程工具从"聊天辅助"升级为"自动化工程系统"的团队
- 需要管理多模块、多语言大型项目的开发者
- 希望减少人工干预、提升 AI 辅助开发可靠性的场景

## 详细内容

> 本文整理自 Anthropic 工程师 Boris Cherny 审阅的 69 条建议，浓缩为 12 个核心模式。这些能力（Agents、Skills、Hooks、Worktrees、Loops）Claude Code 均已内置，大多数人只是从未发现它们。

---

### 实践 1：优化 CLAUDE.md 文件结构

**问题**：CLAUDE.md 过长（如 500+ 行）会导致后半部分规则被 Claude 悄悄降权忽略。

**最佳实践**：
- 60 行最优，200 行为上限
- 按目录分层管理规则，而非单文件堆积

**推荐目录结构**：
```
my-service/
├── CLAUDE.md           # 根目录：通用规则，约 30 行
├── api/
│   └── CLAUDE.md       # REST 约定、错误码
├── internal/
│   └── CLAUDE.md       # 包命名、接口模式
└── cmd/
    └── CLAUDE.md       # 参数解析、日志初始化
```

**条件规则语法示例**（Claude Code 支持）：
```markdown
<important if="language=go">
- 不要裸 goroutine，使用 errgroup 或 WaitGroup 管理生命周期
- 通道容量 > 0 需要行内注释说明原因
- context 必须是第一个参数，命名为 ctx
</important>
```

> 💡 规则只在相关时才出现，而非全量加载到上下文。

### 实践 2：将 Claude Code 当编排系统而非聊天界面

**问题**：每次对话重复写相同的约束提示词，效率低且不可靠。

**解决方案**：使用 `.claude/` 目录进行系统化配置，而非依赖临时提示词。

**.claude/ 三层架构**：

| 目录 | 类型 | 作用 |
|------|------|------|
| `commands/` | 斜杠命令 | 可复用的工作流快捷方式 |
| `agents/` | 子 Agent | 专用 Agent，工具访问权限受限 |
| `skills/` | 知识模块 | 跨项目移植的知识模块 |

**杀手级用法**：在 `.claude/agents/` 中创建只读 `reviewer.md`，该 Agent **物理上无法修改文件**，彻底替代"只读文件，别改任何东西"这类提示词约束。

---

### 实践 3：用 `context: fork` 隔离上下文

**问题**：Skills 默认共享主 Agent 上下文，大量文件读取后会拖慢后续所有任务。

**解决方案**：在 skill 配置中添加 `context: fork`，Skill 在隔离子 Agent 中运行，完成后上下文自动丢弃，主 Agent 只接收干净的输出摘要。

**配置示例**：
```yaml
context: fork
```

**效果**：
- ✅ 无需手动 `/compact` 或重开 session
- ✅ 适合代码质量分析、大批量文件读取等重型 skill

---

### 实践 4：善用 Hooks 实现旁路自动化

Hooks **在 Claude 主推理循环之外执行**，不消耗 token，不中断任务。

| 钩子类型 | 触发时机 | 典型用途 |
|----------|----------|----------|
| `PreToolUse` | 工具执行前 | 拦截危险操作，要求确认 |
| `PostToolUse` | 工具执行后 | 自动格式化、日志记录 |
| `Stop` | 任务结束时 | 完成验证（见实践 11） |

**具体用法示例**：
- **写入后自动格式化**：PostToolUse 钩子在每次写入 Go 文件后自动运行 `gofmt`
- **防护式破坏性操作**：`/careful` 模式激活 PreToolUse 钩子，拦截文件删除、覆盖、强制推送等不可逆操作

---

### 实践 5：用 `/sandbox` 减少 84% 的权限弹窗

**问题**：默认每个 shell 命令都需要手动批准，频繁中断工作流。

**解决方案**：使用 `/sandbox` 在隔离环境中运行命令，减少 **84%** 的权限弹窗。

> ⚠️ 沙盒命令的系统访问有限，适用于大多数常规编码任务。

---

### 实践 6：用 `claude -w` + tmux 并行运行多个 Agent

**核心命令**：`claude -w`——在 git worktree 内启动 Claude，每个实例有独立分支和上下文。

**配合工具**：tmux 多面板管理，适合大型重构、多模块并行开发。

**并行工作示例**：
```
Agent A → 重构 auth 模块（独立分支）
Agent B → 修复数据库层（独立分支）
Agent C → 为 A、B 编写测试（独立分支）
```

> 💡 这是"跟风编程"与"Agent 工程"之间**最显著的差异**：从一次等一个回复，到同时运行十几个 Agent。

---

### 实践 7：`ultrathink` 触发深度推理模式

**使用方式**：在任务描述中任意位置加入 `ultrathink` 关键词，触发扩展思考模式，进行更高努力的推理。

**适用场景**：架构设计、服务拆分、复杂决策等高价值任务。无需配置更改或模型切换。

> ⚠️ 大多数日常任务不需要，建议仅在复杂决策场景使用。

---

### 实践 8：用 `/loop` 实现本地持续自动化

**命令格式**：`/loop <时间间隔> <工作流命令>`

**示例**：`/loop 30m /code-review`——每 30 分钟自动执行一次代码审查。

**与 CI/CD 的核心区别**：

| 对比维度 | CI/CD | `/loop` |
|----------|-------|---------|
| 触发时机 | 代码合并后 | 开发过程中 |
| 上下文感知 | 只看 diff | 知道整个 session 的工作历史 |
| 访问状态 | 仓库快照 | 当前工作状态 |

支持最长间隔 1 小时，最长持续 3 天。

---

### 实践 9：用 `/btw` 提出旁路问题不中断主任务

**命令格式**：`/btw <你的问题>`

**工作机制**：问题加入队列，Claude 在自然暂停点处理后继续主任务。适合长时间重构中途想问不相关问题，不想丢失当前上下文的场景。

---

### 实践 10：Skills 文档中最重要的是 Gotchas（踩坑记录）

**核心建议**：先写 Gotchas 部分，每次 Claude 失败就更新。

**为什么重要**：踩坑记录是具体的、可验证的、可叠加的，信噪比远高于解释性文字。

**Gotchas 的价值**：
- 📌 记录 Claude 在实际使用中遇到的**真实失败模式**
- 📌 不是预期的边界情况，而是**已发生的失败**
- 📌 每次记录的失败都成为未来的预防

---

### 实践 11：用 Stop 钩子让"完成"真正代表完成

**问题**：Claude 声称完成，但实际可能存在空错误处理、缺失文件等问题。

**解决方案**：配置 Stop 钩子连接验证脚本，验证内容包括：
- 运行测试套件
- 检查文件存在性
- 调用 API 端点验证响应

验证失败则任务状态回滚，Claude 继续工作直到通过。适合跨夜运行、多 session 自动化工作流。

---

### 实践 12：`--bare` 参数让 SDK 调用快 10 倍

**适用场景**：通过 Agent SDK **编程方式**调用 Claude（非交互式 CLI）。

**命令示例**：`claude --bare --prompt "analyze this file"`

**原理**：跳过上下文发现（定位 CLAUDE.md、加载配置、检查环境）的启动过程，启动时间减少最高 **10 倍**。

**实测数据**：
```
场景：300 个文件，每个文件一个 Agent 并行运行
配置前：约 40 分钟
配置后：约 18 分钟
节省：55% 的时间
```

> ⚠️ 单次调用效果不明显，批量自动化场景收益显著。

---

### 总结对比：跟风编程 vs Agent 工程

| 维度 | 跟风编程 | Agent 工程 |
|------|----------|------------|
| **约束来源** | 每天手写提示词 | 配置文件 + 权限限制 |
| **工作方式** | 描述→尝试→纠正→重复 | 预设规则→自动执行→自动验证 |
| **上下文管理** | 手动 `/compact` | `context: fork` 自动清理 |
| **权限控制** | 每次手动批准 | Agent 物理上无权做某些操作 |
| **可复用性** | 每次重写提示词 | Skills / Commands / Agents |
| **并行度** | 一次一个对话 | 几十个 Agent 同时运行 |
| **验证方式** | 肉眼检查 | Stop 钩子自动验证 |

> 📌 **核心洞察**：跟风编程和 Agent 工程之间的真正差距，**不在于不同的工具，而在于同样的工具以不同的方式使用**。
