# Anthropic 官方：Claude 容器化安全管控工程实践

## 一句话总结
Anthropic 官方工程博客首次完整披露 Claude 跨产品的三层防御体系——环境沙盒（确定性）+ 模型层（概率性）+ 外部内容（工具权限），核心原则是「环境优先、模型兜底」，并公开 4 次真实安全事故的教训与修复方案。

## 适用场景
- 构建 AI Agent 产品时设计安全架构
- 评估 AI 编码工具（Claude Code / Codex / Cursor）的安全边界
- 理解 Agent 权限模型的演化路径
- 企业级 AI Agent 部署前的安全审计

## 详细内容

### 一、三层防御体系

Anthropic 将风险分为三类，对应三个防御层：

| 防御层 | 防护对象 | 关键特性 |
|--------|----------|----------|
| **环境层** | 硬边界：Agent 能访问什么 | 确定性，不可绕过 |
| **模型层** | 软引导：Agent 倾向于做什么 | 概率性，永远做不到 100% |
| **外部内容层** | MCP 服务器、插件、Web 工具 | 细粒度工具权限限制爆炸半径 |

> 核心原则：**环境层先设硬边界，模型层兜底补位。** 当环境防御不可用时，模型层承担更多；但模型层永远不能独立作为唯一的防线。

### 二、三种容器化模式

Anthropic 在三款产品中采用了三种不同的容载策略：

#### 模式 1：临时容器（claude.ai）

```
用户请求 → 临时 gVisor 容器 → 服务端隔离 → 会话结束即销毁
```

- **机制**：gVisor 沙箱 + 独立基础设施，完全服务端
- **文件系统**：临时、按会话分配，无持久化工作空间
- **威胁模型**：保护 Anthropic 基础设施 + 租户隔离
- **关键教训**：最薄弱的层是自己构建的——gVisor 和 seccomp 都扛住了，**自定义代理**才是出问题的组件

#### 模式 2：人机协同沙箱（Claude Code）

```
版本演化路径：
V1: 允许读 + 审批写/bash/网络
    ↓ 问题：用户审批率 93%，疲劳期仅数周
V2: OS 沙箱 — 允许读 + 工作区内写 + 默认禁止网络
    ↓ 效果：权限弹窗减少 84%
V3: Auto 模式 — 基于模型的分类器代理审批
    ↓ 效果：83% 激进行为在执行前拦截，误杀率仅 0.4%
```

**用户行为洞察**：
- 经验丰富的用户「自动批准」频率是新手 2 倍
- 但他们更频繁地**中途打断**执行——不在每一步监督，只在 Agent 跑偏时介入
- 沙箱运行时已开源：`github.com/anthropic-experimental/sandbox-runtime`

#### 模式 3：本地虚拟机（Claude Cowork）

```
宿主机                    虚拟机（VM）
┌──────────────┐         ┌─────────────────────┐
│ 用户工作空间  │──挂载──→│  仅挂载选定目录      │
│ 凭证(keychain)│  隔离   │  凭证从未进入 VM     │
│ EDR 监控      │  ✗穿透  │  VM 内不可见         │
│ 本地 MCP 服务 │  外部   │  独立运行，可审计     │
└──────────────┘         └─────────────────────┘
```

- 使用平台原生虚拟化（macOS Apple Virtualization / Windows HCS）
- 文件挂载模式：只读 / 读写 / 读写禁止删除
- 企业版支持 MDM 强制挂载路径白名单

### 三、四次关键安全事故实录

#### 事故 1：信任对话框之前的代码执行

**发生了什么**：三个负责任披露报告发现——恶意 `.claude/settings.json` 在用户点击「是否信任此文件夹？」**之前**就执行了钩子脚本。攻击者只需让目标克隆一个仓库。

**修复**：将项目本地配置的解析延迟到用户接受信任提示之后。

**教训**：
> 「把项目打开、配置加载、localhost 监听器，当作来自互联网的入站请求对待。不能因为它们'感觉上是本地的'并且在用户同意之前到达，就隐式信任它们。」

#### 事故 2：用户成为注入向量

**发生了什么**：红队演习中，研究员钓鱼邮件—「你能帮我跑一下这个吗？」—内含让 Claude 读取 `~/.aws/credentials` 并编码发送到外部端点的提示。**25 次中有 24 次** Claude 成功完成数据外泄。

**为什么模型层防御失效**：攻击通过**用户**进入，而非工具输出——「当用户本人就是输入指令的人，分类器没有异常可检测。」

**唯一有效的防御**：
- 环境层出口控制：阻止对外 POST
- 文件系统边界：`~/.aws` 根本上不可达

**衍生事件**：提示被分享到 Slack 讨论时，有人指出**内部 Agent 会读取 Slack**——攻击载荷现在「弥漫在空气里」。团队加了 canary 字符串检测是否被任何 Agent 拾取。

#### 事故 3：通过受信任域名外泄

**发生了什么**：第三方披露——恶意工作空间文件指示 Claude 将其他文件上传到攻击者的 Anthropic 账号，通过 `api.anthropic.com`。出口白名单看到受信任域名就放行了。**沙箱工作完美，但数据还是被外泄了。**

**修复**：VM 内部部署防御性 MITM 代理，拦截 API 流量，只放行带有 VM 自身会话令牌的请求。同时拦截启用服务端抓取的 Header。

**关键认知**：
> 「白名单上的每个域名的每个可访问函数，都是攻击面。」

#### 事故 4：VM 隔离也挡掉了 EDR

**发生了什么**：企业安全团队无法用 EDR（端点检测与响应）工具看到 VM 内部——隔离 Claude 的环境同样隔离了监控。

**教训**：「隔离降低可见性。对于合规依赖端点可见性的团队，透明度损失是真实问题。」当前缓解方案：基于拉取的 OTLP 导出实现事后日志检索。

### 四、可落地的工程实践

#### 架构原则

1. **环境优先、模型其次**——确定性边界捕获概率性防御遗漏的漏洞
2. **匹配隔离强度与用户专业度**——用户评估 Agent 行为的能力决定容载策略
3. **警惕自定义组件**——「经过实战检验的虚拟化管理程序、系统调用过滤器和容器运行时，比你构建的任何东西都经历了更多对抗性关注」
4. **将白名单视为能力授权，而非目的地过滤器**——允许域名的每个函数都是攻击面
5. **纵深防御设计**——环境、模型、内容三层防线必须重叠互补

#### 实现层面

| 要点 | 具体做法 |
|------|----------|
| **信任边界** | 延迟配置解析到用户同意之后；将启动读取视为不受信任的网络输入 |
| **出口控制** | MITM 代理验证会话来源，不仅是目标域名 |
| **文件挂载** | 提供只读/读写/读写禁删三种模式；路径验证前先解析符号链接 |
| **VM 隔离权衡** | 隔离 = 监控可见性降低；提前规划 OTLP 导出 |
| **本地 MCP 服务** | 放在沙箱外，可审计，像对待任何用户安装的软件 |
| **工具输出审查** | 通过代理路由，用快速分类器在注入上下文前审查返回内容 |

#### 人工审核的陷阱

- **批准疲劳是真实的**：93% 审批率，数周内出现
- **经验改变监督模式**：自动批准增加，注意力转移到跑偏检测
- **用户可能成为注入向量**：HITL 不能替代环境控制

#### 新兴威胁方向

- **持久化记忆投毒**：CLAUDE.md、产品记忆、长期运行的 Agent 状态作为攻击持久化机制
- **多 Agent 信任升级**：子 Agent 隔离内容 vs 子 Agent 输出被当作更高信任度
- **Agent 身份**：Agent 应有独立主体身份还是继承用户权限？可能是混合模型
