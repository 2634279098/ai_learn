# Anthropic 构建高效 AI Agent 指南

## 一句话总结
Anthropic 基于与数十个团队构建 LLM Agent 的实战经验，提出了以"简单性、透明性、工具设计"为核心的三大原则，并系统梳理了六大工作流模式和工具使用最佳实践，帮助开发者构建可靠、可控的 AI Agent 系统。

## 适用场景
- 需要构建生产级 AI Agent 系统，但不确定从何入手
- 希望了解 Anthropic 官方推荐的 Agent 架构设计和最佳实践
- 正在使用 Claude API 构建 Agent，遇到工具调用不稳定、效果不如预期的问题
- 需要在 Workflow（确定性流程）和 Agent（自主决策）之间做技术选型

## 详细内容

### 一、Agent 的定义与分类

#### 关键区别
- **Workflows（工作流）**：通过预定义代码路径编排 LLM 和工具，适合明确定义的任务
- **Agents（代理）**：LLM 动态控制自己的流程和工具使用，保持对任务完成方式的控制权，适合需要灵活性的场景

#### 使用建议
- **优先使用最简单的解决方案**，只有在需要时才增加复杂性
- Agent 系统通常用延迟和成本换取更好的任务性能
- **Workflows** 适合明确定义的任务（可预测性和一致性优先）
- **Agents** 适合需要灵活性和模型驱动决策的场景

---

### 二、六大工作流模式

#### 1. Prompt Chaining（提示链）
**原理**：将任务分解为一系列步骤，每个 LLM 调用处理前一个的输出

**适用场景**：
- 任务可以清晰分解为固定子任务
- 用延迟换取更高准确性

**实例**：
- 生成营销文案 → 翻译成其他语言
- 写文档大纲 → 检查大纲 → 基于大纲写完整文档

---

#### 2. Routing（路由）
**原理**：对输入进行分类，导向专门的后续任务

**适用场景**：
- 复杂任务有不同类别需要分别处理
- 分类可以准确处理（通过 LLM 或传统分类模型）

**实例**：
- 将不同类型的客服查询（常规问题、退款请求、技术支持）导向不同流程
- 简单问题路由到 Claude Haiku（高效低成本），复杂问题路由到 Claude Sonnet（高性能）

---

#### 3. Parallelization（并行化）
**原理**：LLM 同时处理任务，程序化聚合输出

**两种变体**：
- **Sectioning（分块）**：将任务分解为独立子任务并行运行
- **Voting（投票）**：多次运行同一任务获取多样化输出

**适用场景**：
- 子任务可并行化以加速
- 需要多视角或多次尝试以提高置信度

**实例**：
- **Sectioning**：一个模型处理用户查询，另一个筛查不当内容
- **Voting**：多个提示从不同角度审查代码漏洞

---

#### 4. Orchestrator-Workers（编排器-工作器）
**原理**：中央 LLM 动态分解任务，委托给工作 LLM，综合结果

**适用场景**：
- 无法预测需要的子任务（如编码时，需要变更的文件数和性质取决于具体任务）
- 与并行化的关键区别：**灵活性** - 子任务不是预定义的，而是由编排器根据特定输入决定

**实例**：
- 需要对多个文件进行复杂变更的编码产品
- 需要从多个来源收集和分析信息的搜索任务

---

#### 5. Evaluator-Optimizer（评估器-优化器）
**原理**：一个 LLM 调用生成响应，另一个在循环中提供评估反馈

**适用场景**：
- 有明确的评估标准
- 迭代优化能提供可测量的价值
- LLM 响应可以通过人类反馈显著改进
- LLM 能提供此类反馈

**实例**：
- 文学翻译，评估器 LLM 提供有用的批评
- 需要多轮搜索和分析的复杂搜索任务

---

#### 6. Agents（自主代理）
**原理**：LLM 成熟到理解复杂输入、参与推理和规划、可靠使用工具、从错误中恢复

**工作流程**：
1. 从人类用户的命令或交互讨论开始
2. 任务明确后，代理独立规划和操作
3. 可能返回人类获取更多信息或判断
4. 每一步从环境获得"真实反馈"（工具调用结果或代码执行）
5. 可在检查点暂停等待人类反馈
6. 通常完成时终止，也可设置停止条件（如最大迭代次数）

**注意事项**：
- 自主特性意味着更高成本和潜在的复合错误
- 建议在沙箱环境中广泛测试，并配备适当的防护措施

### 三、工具使用最佳实践（Agent-Computer Interface）

**核心原则**：就像重视人机界面（HCI）一样，要同样重视**代理-计算机界面（ACI）**的设计。Anthropic 发现，构建 SWE-bench Agent 时花费了**更多时间优化工具而不是整体提示**。

#### 工具格式建议
1. **给模型足够的 token 来"思考"**，避免写进死角
2. **保持格式接近模型在互联网文本中自然遇到的格式**
3. **确保没有格式"开销"**，如需要准确计数数千行代码或转义字符串

#### 具体实践建议

**像写优秀文档字符串一样优化工具描述**：
- 使用清晰的参数名称和描述
- 包含示例用法、边缘情况、输入格式要求
- 明确与其他工具的清晰边界

**测试模型如何使用工具**：
- 在 workbench 中运行许多示例输入，观察模型犯的错误，迭代改进

**Poka-yoke（防错）你的工具**：
- 更改参数使犯错更困难
- **实际案例**：发现模型在代理移出根目录后会对使用相对文件路径的工具犯错 → 修复方法：更改工具使其始终需要绝对文件路径 → 结果：模型完美使用这个方法

---

### 四、三大核心原则

文章总结的成功构建 Agent 的三大核心原则：

1. **Simplicity（简单性）**：保持 Agent 设计简单，从简单提示开始
2. **Transparency（透明性）**：通过明确显示 Agent 的规划步骤来实现可观测性
3. **Tool Documentation and Testing（工具文档和测试）**：通过彻底的工具文档和测试来精心设计代理-计算机界面（ACI）

**成功路径**：
```
简单提示 → 通过全面评估优化 → 只在更简单的解决方案不足时，才添加多步骤 Agent 系统
```

### 五、实际应用场景

#### A. 客户支持
**为什么适合 Agent**：
- 支持交互自然遵循对话流程，同时需要访问外部信息和操作
- 可以集成工具拉取客户数据、订单历史、知识库文章
- 可以程序化处理退款或更新工单等操作
- 成功可以通过用户定义的解决方案明确衡量

**商业模式创新**：一些公司通过基于使用的定价模式（仅对成功解决方案收费）证明了这种方法的可行性。

#### B. 编码 Agent
**为什么有效**：
- 代码解决方案可以通过自动化测试验证
- Agent 可以使用测试结果作为反馈迭代解决方案
- 问题空间定义良好且结构化
- 输出质量可以客观衡量

**当前成果**：Agent 现在可以仅基于 pull request 描述解决 SWE-bench Verified 基准测试中的真实 GitHub 问题。

---

### 六、框架使用建议

#### 推荐框架
- **Claude Agent SDK**（官方）
- **AWS Strands Agents SDK**
- **Rivet**（拖放式 GUI LLM 工作流构建器）
- **Vellum**（用于构建和测试复杂工作流的 GUI 工具）

#### 使用建议
1. **建议开发者直接从 LLM API 开始**：许多模式可以用几行代码实现
2. **如果使用框架，确保理解底层代码**：错误的假设是客户错误的常见来源
3. **框架可以加速入门，但在生产环境考虑减少抽象层，用基础组件构建**

---

### 七、总结建议

> **成功的关键不是构建最复杂的系统，而是构建适合需求的*正确*系统。**

具体建议：
1. 从简单提示开始
2. 通过全面评估优化它们
3. **只有在更简单的解决方案不足时，才添加多步骤 Agent 系统**

这些构建块不是规定性的，而是开发者可以根据不同用例塑造和组合的常见模式。成功的关键是**衡量性能并迭代实现**。
