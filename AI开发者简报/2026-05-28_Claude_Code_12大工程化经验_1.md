# Claude Code 12 大工程化经验

## 一句话总结
从"能跑"到"好用"，将 Claude Code 从简单的 AI 聊天工具升级为工程化的 Agent 开发平台，12 条实战经验覆盖配置管理、自动化、成本控制、安全防护全链路。

## 适用场景
- 已安装 Claude Code 但只用了基础功能，想深度挖掘工具潜力的开发者
- 需要将 Claude Code 集成到团队工作流中的项目负责人
- 关注 AI 编程工具降本增效的技术管理者

## 详细内容

### 一、写一份真正有用的 CLAUDE.md

**本质**：`CLAUDE.md` 不是 README，是每次会话自动注入上下文的高优先级指令文件。

**好的写法**：
- 技术栈声明（语言版本、框架、数据库）
- 代码风格约定（命名规范、缩进、注释风格）
- 错误处理规范
- "必须先看的文件"列表
- 明确禁止事项（如禁用 `print()` 调试、禁止 `from x import *`）

**坏的写法**：
- 写项目背景、团队规模、技术选型理由——这些是给人看的，对模型是噪声

**经验值**：控制在 **200 行以内**，超过部分用 `@import` 拆到子文件按需引入。

---

### 二、用 Hooks 把重复提醒自动化

Hooks 是"杀手锏"功能，可在工具调用前后自动插入 shell 脚本，将规则从对话提示降级为强制硬约束。

**配置位置**：`~/.claude/settings.json`

| Hook 类型 | 作用 | 示例 |
|-----------|------|------|
| **PostToolUse** | 工具调用后执行 | Edit/Write 完 Python 文件后，自动跑 `ruff format` 和 `ruff check --fix` |
| **PreToolUse** | 工具调用前拦截 | 检测危险命令（`rm -rf`、`git push --force`、`drop database`），匹配到直接阻止执行 |

**核心价值**：不用在 prompt 里反复提醒，模型也无法"忘记"。

---

### 三、自定义 Slash Command 封装高频流程

适用于每周重复执行的操作（写 changelog、生成 API 文档、issue 转 PR description）。

**实现方式**：在 `.claude/commands/` 目录下创建 Markdown 文件：

```markdown
---
description: 根据最近的 git 提交生成 CHANGELOG 条目
---
读取从上一个 tag 到 HEAD 的所有 commit，按 feat/fix/chore/docs 分类...
```

调用方式：`/changelog`

**进阶用法**：支持参数传递，例如 `/review 1234`，通过 `$ARGUMENTS` 变量获取参数值。

---

### 四、并行 Subagent 压缩任务时间

Claude Code 的 Agent 工具可一次启动多个 subagent，每个独立上下文并行运行。

**适用场景**：
- 在 5 个不同模块同时搜索 API 调用点
- 给 10 个 service 类同时加同一种 logging
- 同时跑代码 review、安全扫描、性能检查

**实测效果**：串行 90 秒的任务，三个并行可压缩到 30 秒。

---

### 五、Plan Mode 防止"边想边改"翻车

**触发方式**：`Shift+Tab` 切换，或说"先给我一份实施计划，不要动代码"。

**特别适合的场景**：
- 涉及 3 个以上文件的重构
- 引入新依赖或新架构模式
- 数据库 migration / API 版本变更
- 不熟悉的代码区域

**流程**：Claude 先探索代码 → 列出方案 → 给出步骤清单 → 用户确认后再动手。

---

### 六、上下文管理：长会话要主动压缩

**核心原则**：上下文窗口虽然大（Opus 4.7 支持 1M），但不等于无脑塞。

| 方法 | 用途 |
|------|------|
| `/clear` | 完成子任务后清空上下文，下一个任务从干净状态开始 |
| `/compact` | 把前面会话压成摘要，保留关键决策、去掉冗余 |
| 写进 CLAUDE.md | 跨会话的重要决策不依赖"上次会话我们决定了 X" |

**风险提示**：会话超几千轮后，早期无关讨论会污染模型注意力，token 费用持续累加。

---

### 七、让模型直接读 Issue / PR / Log

让 Claude 直接读源头数据，比手动复制粘贴更准确。

**典型用法**：
- 读 Issue：`gh issue view 456 --comments`
- 分析 CI 失败：`gh run list -L 1` + `gh run view <id> --log-failed`
- 分析生产日志：直接读取日志文件最后 N 行，按 ERROR 级别归类

---

### 八、`/cost` 命令让账单可见

**核心功能**：显示当前会话累计费用。

**建议习惯**：
- 每个大任务前后各查一次 `/cost`，评估单任务成本
- 上下文膨胀到几十万 token 时能看到费用拐点
- 团队协作时以会话费用评估"这个功能开发用了多少 AI 成本"

---

### 九、用 Skills 沉淀团队最佳实践

Skills 比 slash command 更结构化，可带说明文件、模板、辅助脚本。

**示例结构**（新建微服务 Skill）：
```
.claude/skills/new-microservice/
├── SKILL.md              # 描述 + 触发条件
├── templates/
│   ├── pyproject.toml.tpl
│   ├── Dockerfile.tpl
│   └── main.py.tpl
└── scripts/
    └── setup.sh
```

**触发机制**：当用户说"新建一个 XX 服务"时，模型自动匹配并执行统一规范。

---

### 十、模型分层：贵的留给硬骨头

**核心策略**：不是所有任务都需要最强模型。

| 任务类型 | 推荐模型 | 理由 |
|----------|----------|------|
| 复杂重构、架构设计、跨多文件逻辑 | `claude-opus-4-7` | 最强推理能力 |
| 写新功能、改 bug、单文件改动 | `claude-sonnet-4-6` | 速度快 5 倍，质量足够 |
| 简单查找、文档撰写、commit message | `claude-haiku-4-5` | 价格 1/20，秒回 |

**效果**：默认 Sonnet 4.6，只在硬骨头时切 Opus，月成本可下降 **60% 以上**。

---

### 十一、Permission 配置：减少弹窗、保留安全

**配置位置**：`~/.claude/settings.json` 的 `permissions` 字段。

| 策略 | 示例 |
|------|------|
| **默认放行**（高频安全操作） | `npm test`、`git status/diff/log`、`pytest`、读文件、编辑 `src/` |
| **默认拒绝**（危险操作） | `rm -rf`、`git push --force`、`curl \| sh`、编辑 `.env` 或 `secrets/` |

---

### 十二、稳定接入：排除网络变量

**问题**：Claude Code 直连 Anthropic 官方 API 在国内可能请求中断、响应卡顿。

**解决方案**（环境变量）：
```bash
export ANTHROPIC_AUTH_TOKEN="sk-你的密钥"
export ANTHROPIC_BASE_URL="https://gw.claudeapi.com"
```

---

### 最优先做的三件事

1. **写一份精简但具体的 CLAUDE.md**（不超过 200 行）
2. **启用"自动格式化 + 危险命令拦截" hooks**
3. **默认模型设为 Sonnet 4.6**，只在硬骨头时切 Opus 4.7
