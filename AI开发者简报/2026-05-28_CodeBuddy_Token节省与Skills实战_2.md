# CodeBuddy Token 节省与 Skills 实战

## 一句话总结
鹅厂高工分享的 AI 编程 Token 成本优化方法论：通过 Rules 约束输出、Skills 按需加载上下文、模型分层使用、Prompt Caching 四大手段，将 AI 编程 Token 消耗降低 50% 以上。

## 适用场景
- 使用 CodeBuddy 等 AI 编程工具的开发者，发现 token 费用超出预期
- 需要建立团队级 AI 编程规范，统一降低上下文成本的团队
- 希望理解 Agent Skills 工程化原理和 Token 省钱的底层逻辑

## 详细内容

### Token 消耗的两大真相

1. **Input Token 的"滚雪球"效应**：大模型无状态，每次请求需将历史对话、引用文件、系统提示词全部打包发送。对话轮次越多，单次请求的 Input 成本越高。
2. **Output Token 比 Input 贵 3-4 倍**：AI 生成的"废话"（如"好的，我明白了..."）是最昂贵的浪费。

**降本核心思路——提高信噪比**：
- **输出端**：禁止 AI 生成昂贵的废话
- **输入端**：控制上下文长度，只喂必要信息
- **处理端**：根据任务难度分级，不让"博士生做算术题"

---

### 技巧一：制定 Rules（规则）

**痛点**：AI 会"热心"附赠注释、文档、测试用例、复述需求等无效输出。

**对策**：在 Rules 中写入硬性约束，建立防御边界。

**可直接复用的规则模板**：

| 类别 | 具体规则 |
|------|---------|
| **输出约束** | 严禁生成 README、文档说明、功能总结；默认禁止生成测试用例；禁止复述需求 |
| **交互风格** | 零客套（禁止"好的""我明白"等填充词）；直接输出代码；只给唯一最优解 |
| **分场景解释** | 简单修改→零解释；新增功能→1-2句简述；关键决策→必须简要说明理由 |
| **执行原则** | 最小修改，禁止自作主张重构；歧义时仅提 1 个关键确认问题 |

**操作**：在 CodeBuddy 设置中添加 Project Rule，粘贴规则内容。注意内容要**字字珠玑**，过长的 Rules 反而会干扰模型。

---

### 技巧二：需求拆分（一次性说清）

**痛点**："挤牙膏"式对话导致 AI 反复生成代码，Token 消耗指数增长。

**对策**：一次性把要求、边界条件、参考范例说清楚。少说"不要做什么"，多说"要怎么做"并给出**正确示范**。

**Prompt 示例结构**：
- 业务背景
- 数据定义（字段、来源、单位、默认值）
- 路由注册（参考具体页面）
- UI 规范（布局、组件）
- 组件规范（表单/列表使用的具体库）
- 逻辑参考（权限控制方式）

**Tips**：专有技术名词用英文（如 `debounce function`、`Dependency Injection`），复杂逻辑解释可用中文。

---

### 技巧三：技术方案确认（先对齐再动手）

**痛点**：AI 理解偏差导致生成大量错误代码，不仅浪费 Token，还可能覆盖未提交代码。

**对策**：让 AI 先用自然语言描述修改计划、涉及文件、核心逻辑，**确认无误后再让它生成代码**。

**踩坑案例**：作者遇到 `goosefsx` 和 `goosefs` 两词被 AI 混淆，全局多个模块被改得面目全非。如果先确认方案，这个错误根本不会发生。

---

### 技巧四：上下文长度控制（五种方法）

#### 4.1 手动控制上下文
- **改哪里，选哪里**：不要让 AI 扫描整个项目
- 选中代码 → 添加引用，或使用 `@Files` 指定文件

#### 4.2 合理使用多对话框
- **专人专事，用完即走**：不要把 AI 当永远不关机的聊天窗口
- 聊方案细节可用便宜模型，确定方案后新建对话框交给高级模型
- 大需求拆成多个小需求，每个小需求一个**新对话框**

#### 4.3 压缩上下文
- 在 CodeBuddy 设置中新建指令 `/compact`，内容设为"总结当前进度，压缩上下文"

#### 4.4 清空上下文
- AI 开始"左右互搏"、逻辑混乱时，不要试图纠正，直接**清空上下文**或**新建对话**

#### 4.5 按需加载上下文（Agent Skills）
- 大模型 = **全科医生**（处理通用对话、逻辑分析）
- Skills = **专科医疗包**（针对特定领域封装的专用 Prompt、规则和示例）

### Agent Skills 深度解析

#### 为什么要按需加载？

| 问题 | 说明 |
|------|------|
| **Token 浪费** | 所有规范塞进 System Prompt，写 CSS 也要加载代码合并规则 |
| **注意力稀释（Context Pollution）** | 无关规则干扰模型判断，导致幻觉或指令遵循能力下降 |

**原理**：平时不加载，用时才调用。Agent 识别到特定场景时才动态注入对应上下文，处理完后不污染后续对话。

#### Skills 编写三步法

**第一步**：识别场景——什么样的场景适合做 Skills？
- ✅ 代码冲突解决、生成单元测试、老旧代码重构（流程复杂、规则繁琐、非每次对话都需要）
- ❌ 通用代码命名规范（应放全局 Rules）

**第二步**：利用大模型生成 Skills
> "我需要一个专门解决 Git 代码冲突的 Skills。请帮我生成一个 CodeBuddy 格式的 Skills 配置。根据当前工作区冲突的文件，分析都有哪些种类的冲突，分别给出解决方案。"

**第三步**：配置 Skills
- 文件结构包含 **Metadata（元数据）** 和 **Instructions（指令）**
- Metadata 定义：`name`、`description`、`triggers`、`globs`、`alwaysApply`、`when`
- **注意**：SKILL 文件名要大写

#### 使用 Skills vs 不使用的对比

| 维度 | 不使用 Skills | 使用 Skills |
|------|-------------|------------|
| 上下文 | 全局 Prompt 一直带着几百字规范 | 平时对话上下文干净 |
| 触发 | 每次对话都付费 | 识别到触发词时才加载 |
| 副作用 | 可能错误应用到无关代码 | 解决完后退出，不影响后续 |

### 模型分层策略

**理念**：杀鸡焉用牛刀

| 模型级别 | 适用场景 |
|---------|---------|
| **高级模型（如 GPT-5.2）** | 复杂逻辑分析、架构设计、疑难 Bug 修复 |
| **普通/免费模型** | 写工具函数、定义 TypeScript 类型、生成样板代码、写注释 |

**操作技巧**：让高级模型把接口定义、类型定义、工具函数等简单需求拆分出来，然后手动选择免费模型实现，剩余复杂需求交给高级模型。

---

### Prompt Caching（提示词缓存）

**原理**：如果 System Prompt、Rules 和基础代码库上下文是固定的且放在 Prompt 前缀部分，API 厂商会缓存这部分内容。

**省钱效果**：缓存命中的 Token 价格通常只有写入价格的 **1/10**。

**具体操作**：
1. 确保 Rules 和常用 Reference Files 的**内容和顺序保持不变**
2. **尽量放在对话的最前面**（在对话最开始手动引入 Rules 和 Files，再开始其他提示词）
3. 这样能最大程度触发缓存命中

---

### 优化前后效果对比

| 维度 | 优化前（粗放模式） | 优化后（精细模式） | 预期收益 |
|------|-----------------|-----------------|---------|
| 上下文 | 全局扫描，动辄 50k+ token | 手动引用+及时压缩，控制在 10k 内 | Input 成本降低 **60%-80%** |
| 输出 | 包含大量解释、文档和未修改代码 | 仅输出 Diff 代码和关键注释 | Output 成本降低 **50%+** |
| 准确率 | 上下文污染导致幻觉，反复修改 | 专人专事（Skills），方案先行 | 减少返工带来的隐形浪费 |

---

**一句话心法**：把 AI 当作按字数收费的顶级外包专家——**需求给准、资料给对、废话不说**。
