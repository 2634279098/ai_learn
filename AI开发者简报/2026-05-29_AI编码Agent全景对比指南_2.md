### 三、三大代理循环架构对比

#### 1. Plan/Act 分离（Cline、Roo、OpenCode）

```
模型先发出计划（无文件变更）→ 用户批准/编辑 → 切换到 Act 模式执行
```
- **优势**：可审计，适合生产代码和数据库迁移
- **劣势**：较慢，长任务中确认成为瓶颈

#### 2. Architect/Editor 配对（Aider）

```
推理模型 → 英文描述变更 → 快速编辑模型 → 精确 diff
```
- **优势**：大型重构最可靠，规划者不浪费 token 在语法上
- **劣势**：每轮成本更高

#### 3. 子代理和编排器（Claude Code、Kilo Code、Grok Build）

```
顶层代理 → 派生专业子代理（测试运行器/Schema 迁移/前端样式）→ 各自有上下文窗口
```
- **优势**：并行处理复杂任务
- **劣势**：可调试性差，子代理内部出错时可见性低于平面 trace

### 四、头部工具深度解析

#### Claude Code 2.1 — 重构之王

- SWE-bench Verified: **87.6%**（排名第一梯队）
- Opus 4.7 原生上下文扩展到 **1M tokens**
- 90k 行代码库 76 文件重构仅需 **4 小时/3 会话**
- 插件市场作为一等公民系统（2026 春季开放）
- **短板**：仅支持 Anthropic 模型，Pro 版速率限制严格

#### Cursor 3.5 + Composer 2.5 — 成本效益之星

- Composer 2.5 自研编码模型，每任务成本 **$0.07（标准）/ $0.44（快速）**
- 比 Opus 4.7 和 GPT-5.5 便宜 **10-60 倍**
- Artificial Analysis 编码代理指数: **62 分**（第三名）
- Agents Window 支持多仓库并行运行
- **短板**：高强度代理模式下信用池耗尽快，降级后质量不一致

#### Cline 3.85 — 开源审计首选

- 完全开源（Apache 2.0），61K+ GitHub 星标
- 人机协同确保每次操作可追溯
- v3.85 新增 GPT-5.5、DeepSeek V4 Flash/Pro、Gemini 3.5 Flash
- **短板**：人机确认在长任务中成为瓶颈（6 小时 vs Claude Code 4 小时）

#### Grok Build — 并行隔离创新者

- 最多 **8 个子代理**并行，每个在独立 Git worktree 中
- SWE-Bench Verified: **70.8%**（距领先者约 17 个百分点）
- 本地优先设计，零代码库数据发送到 xAI
- **短板**：早期 Beta、$300/月续费价格高昂

### 六、2026 年最佳实践建议

#### 选择决策树

```
代码库超过 50k 行？
├── 是 → Claude Code + Opus 4.7 / Aider Architect / Grok Build
└── 否 → Cursor Tab、Copilot 等内联补全即可

代码能否合法离开机器？
├── 否 → Continue.dev + Ollama / OpenCode / Grok Build（本地优先）
└── 是 → 云方案可选全部工具

需要会话在中断后恢复？
├── 是 → Codex CLI /goal（唯一 GA 的持久线程方案）
└── 否 → 常规工具

谁付账？
├── 公司按席位 → Cursor Business、Windsurf Teams、Claude Code Premium
└── 个人报销 API → Cline、Aider、Roo、Kilo（BYOK 零加价）
```

### 七、成本优化实用技巧

1. **Cursor Composer 2.5**：日常用 $0.07/task 标准模式，只在最难任务用前沿模型
2. **Roo Code diff 编辑**：利用 `apply_diff` 只发送更改行，比 Cline 节省 30% token
3. **Aider Architect/Editor 分离**：规划用前沿模型，编辑用快速便宜模型
4. **混合模型策略**：前沿模型做规划器/架构师位置，本地模型做编辑器位置
