# OpenAI Codex 最佳实践指南

## 一句话总结
OpenAI Codex 提供云端智能体和本地 CLI 两种 AI 编程模式，通过 AGENTS.md 分层配置和三级安全审批机制，让开发者在"行动优先"原则下高效且安全地使用 AI 辅助编码。

## 适用场景
- 需要使用 AI 进行日常编码辅助，希望 AI 直接交付代码而非只给建议
- 团队希望统一 AI 辅助开发的规范和质量标准的场景
- 需要在 CI/CD 流水线中引入 AI 自动化编程能力的场景
- 对 AI 写代码的权限控制和安全隔离有要求的场景

## 详细内容

### 一、OpenAI Codex 两条产品线

#### 1. 云端智能体（codex-1）
- **底层模型**：基于 o3 模型微调的 `codex-1`
- **核心能力**：自主读写代码、运行测试、执行命令行工具，甚至直接提交 Pull Request
- **运行环境**：OpenAI 安全沙箱（独立容器，无网络访问权限）
- **工作模式**：支持交互式对话 + 后台任务（可同时并行派发多个任务）
- **访问计划**：ChatGPT Pro / Business / Enterprise / Plus（2025年6月起开放）
- **定位**：适合"挂后台跑大任务"

#### 2. Codex CLI（开源终端编码助手）
- **底层模型**：`codex-mini-latest`（基于 o4-mini），面向低延迟场景
- **开发语言**：Rust，完全开源
- **运行环境**：本地机器，访问本地文件系统
- **安全模型**：三级审批模式（见下文）
- **定位**：适合"坐电脑前盯着改代码"

| 维度 | 云端智能体 | Codex CLI |
|------|-----------|-----------|
| 运行环境 | OpenAI 云端沙箱 | 本地机器 |
| 网络访问 | 无（隔离） | 取决于本地权限 |
| 代码访问 | GitHub 仓库集成 | 本地文件系统 |
| 开源状态 | 闭源 | 完全开源（Rust） |

### 二、提示工程核心原则

#### ⭐ 1. 行动优先原则（Action Bias）
- 提示应引导模型**直接交付可工作的代码**，而非先列计划再等确认
- 模型应默认做出合理假设并向前推进
- 只有在真正被阻塞（缺少关键信息或存在矛盾约束）时才向用户提问
- **正确写法示例**："接到任务后立即开始工作，合理假设模糊部分，完成后展示结果"
- **错误写法示例**："先列出计划，等确认后再执行"（会严重降低效率）

#### ⭐ 2. 上下文收集策略
- **批量读取**：调用工具前先规划好需要哪些文件，一次性并行读取
- **避免串行探索**：不要一个文件一个文件地逐个查看
- **先搜索后新增**：添加新实现前，先搜索代码库中是否已有类似功能

#### ⭐ 3. 代码质量标准
- **正确性优先于速度**，避免冒险捷径、投机性改动和拼凑式修复
- 遵循代码库现有约定，偏离时需说明理由
- 不添加宽泛的 `try/catch`，错误必须显式传播
- 保持类型安全，避免强制类型断言
- 前端任务：避免模板化设计，追求有辨识度的视觉表达

#### 4. Git 脏工作区处理
- 永远不要恢复不是自己做的改动
- 发现意外更改时立即停下询问用户
- 禁止使用 `git reset --hard` 等破坏性命令

### 三、AGENTS.md 机制详解

类似于 Claude Code 的 `CLAUDE.md` 或 Cursor 的 `.cursorrules`，用于给 AI 注入**项目级上下文和规范**。

#### 分层加载规则
Codex CLI 自动扫描并注入 `AGENTS.md`，遵循**分层覆盖原则**：

| 层级 | 路径 | 适用范围 |
|------|------|---------|
| 全局 | `~/.codex/AGENTS.md` | 所有项目的通用默认行为（语言偏好、通用编码风格） |
| 项目 | 仓库根目录 `AGENTS.md` | 项目级约定（构建命令、测试规范、依赖管理） |
| 模块 | 子目录 `AGENTS.md` | 模块级特殊规则（某个微服务的特定 API 约定） |

- 子目录指令会**覆盖**父目录的同名配置
- 消息以根到叶的顺序注入对话历史

#### 团队 AGENTS.md 建议覆盖内容
1. 项目构建和测试命令
2. 代码风格和命名约定
3. 依赖管理策略
4. Git 工作流规范
5. 常见陷阱和注意事项

### 四、安全模型：三级审批模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **Suggest** | 可读取文件，但所有写操作和命令需确认 | 代码审查、学习 |
| **Auto Edit** | 自动编辑文件，但命令行操作需确认 | 日常开发 |
| **Full Auto** | 全自动，编辑和命令都自动执行 | CI/CD、批量任务 |

#### Full Auto 模式的沙箱机制
- **macOS**：使用 Apple Seatbelt（`sandbox-exec`），文件系统只读白名单 + 完全阻断出站网络
- **Linux**：默认无沙箱，推荐使用 Docker 容器隔离，配合 `iptables`/`ipset` 防火墙脚本

> 💡 Full Auto 模式下，Git 仓库的"可回滚性"是最后一道防线，非 Git 仓库会弹出警告确认。

### 五、高级 API 特性（GPT-5.3 Codex API）

#### 1. 上下文压缩（/compact 端点）
- 压缩对话历史，使长任务不会因上下文溢出而中断
- 兼容 ZDR（Zero Data Retention），返回 `encrypted_content` 加密项
- Token 消耗比逐轮累积更可控

#### 2. Phase 机制（必需项）
- `null`：普通输出
- `commentary`：工作中对用户的进度更新
- `final_answer`：最终完成的交付
- ⚠️ **Phase 是必需项**，不正确保留 phase 元数据会导致显著性能下降
- Phase 字段只能附加在 assistant 消息上，**不要添加到 user 消息**

#### 3. 推理强度选择

| 强度 | 适用场景 |
|------|---------|
| **medium** | 大部分日常开发（推荐默认） |
| **high** | 跨模块重构、复杂需求 |
| **xhigh** | 多服务联调、疑难 bug 排查 |

> 💡 建议先用 medium，遇到推理不足再升级，不要一上来就用 xhigh。

### 六、常见失败模式与调试技巧

| 失败模式 | 表现 | 解决方法 |
|---------|------|---------|
| **过度思考** | 执行第一次有用操作前耗时过长 | 提示中明确要求"立即开始行动" |
| **日志式更新** | 机械汇报状态而非自然协作 | 要求"只在关键节点报告进度" |
| **重复性口癖** | 反复使用"好发现"、"明白了"等填充词 | 在提示中直接禁止这些表达 |

**元提示调试技巧**：在模型回复末尾追加反馈，要求它审视自己的指令并建议改进，生成几次后取共性建议，即可得到有针对性的指令优化方案。

### 七、实际开发工作流建议

#### 渐进式引入策略（推荐路径）

```
第一阶段：Suggest 模式
  → 熟悉 Codex 的代码理解能力和建议质量

第二阶段：Auto Edit 模式
  → 在受控环境下逐步增加信任度，用于日常开发

第三阶段：Full Auto + 沙箱模式
  → 在 CI/CD 流水线或批量任务中启用全自动
```

#### 工具配置最佳实践
- **apply_patch**：优先使用 Responses API 内置方式，与模型训练格式完全一致
- **shell_command**：命令作为单个字符串传递（而非数组），始终填写工作目录
- **并行工具调用**：设置 `parallel_tool_calls: true`，能并行的调用绝不串行
- **工具响应截断**：超长输出截断到约 10k Token，保留头尾两部分

#### 团队落地建议
> **先用 Suggest 模式建立信任，再用 Auto Edit 提效，最后才考虑 Full Auto。AGENTS.md 在全员铺开前，最好先让一两个人试跑一周，把规则调顺了再推广。**
