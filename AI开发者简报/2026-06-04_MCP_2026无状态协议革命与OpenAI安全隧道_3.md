# MCP 2026 无状态协议革命 + OpenAI 安全隧道

## 一句话总结
MCP 发布史上最大规模规范更新（2026-07-28 RC），核心是无状态化架构变革；同期 OpenAI 推出 Secure MCP Tunnel，解决企业 AI Agent 访问内部数据的安全难题——两个独立事件正在重塑 AI Agent 与外部工具的连接方式。

## 适用场景
- 构建可水平扩展的 MCP Server
- 企业级 AI Agent 接入内部数据库/API（无需公网暴露）
- 理解 MCP 协议的演进方向，提前规划架构升级
- 评估 AI 工具生态的技术成熟度

## 详细内容

### 一、MCP 无状态协议革命

#### 1.1 核心变革：告别有状态会话

| 维度 | 旧模型 | 新模型（2026-07-28 RC） |
|------|--------|-------------------------|
| 会话 | 需要 `initialize/initialized` 握手 | **完全无状态**，不需要握手 |
| 路由 | 依赖 `Mcp-Session-Id`，必须粘性会话 | 任何请求可路由到任意服务器实例 |
| 扩展 | 水平扩展困难，需共享会话存储 | 标准负载均衡器直接可用 |
| 客户端信息 | 通过握手传递 | 通过 `_meta` 字段在每个请求中传递 |

**对开发者的直接影响**：MCP Server 可以放在标准轮询负载均衡器后面，不再需要共享会话存储或网关层的深度包检测。

#### 1.2 推动无状态的 6 个关键 SEP

| SEP | 技术变更 | 开发者影响 |
|-----|----------|-----------|
| **SEP-2575** | 移除 `initialize/initialized` 握手 | 通过 `_meta` 传递 `clientInfo` 和 `protocolVersion` |
| **SEP-2567** | 移除 `Mcp-Session-Id` 头 | 实现无状态路由，无需粘性会话 |
| **SEP-2260** | 服务端请求必须关联客户端请求 | 消除无上下文的弹窗，所有服务端动作可追溯到用户触发 |
| **SEP-2243** | 要求 `Mcp-Method` 和 `Mcp-Name` 请求头 | **基础设施关键**：负载均衡器无需解析 JSON-RPC body 即可路由 |
| **SEP-2549** | 工具列表包含 `ttlMs` 和 `cacheScope` | 客户端可缓存工具定义，减少冗余网络调用 |
| **SEP-414** | W3C Trace Context 传播 | 跨 Agent-服务链的分布式追踪 |

#### 1.3 三大新能力

**A. MCP Apps（SEP-1865）：服务端渲染 UI**

```
服务器定义 → Host 在沙盒 iframe 中渲染 → 用户交互 → 经审核/授权管道
```

- 在工具定义中添加 `ui` 对象，指定 `templateUrl` 和 `sandbox: true`
- UI 模板通过 JSON-RPC 与 Host 通信
- 每个用户交互都经过与直接工具调用相同的审核和授权管道
- **应用场景**：可视化 SQL 构建器，用户不再需要在聊天框里敲 JSON

**B. Tasks Extension：长耗时操作**

解决 Agent 等待长时间任务超时的问题：

```json
// 服务端不阻塞，直接返回任务句柄
{"type": "task", "taskId": "abc-123", "status": "running"}

// 客户端轮询直到完成
GET tasks/get?taskId=abc-123
→ {"status": "completed", "result": {...}}
```

**C. 其他升级**
- **OAuth（SEP-2468/837）**：客户端必须验证 `iss` 参数防混用攻击；支持动态客户端注册的 `application_type`
- **JSON Schema 2020-12**：`inputSchema` 和 `outputSchema` 支持 `oneOf`、`anyOf`、`$ref`
- **弃用政策**：Active → Deprecated → Removed，Deprecated 到 Removed 至少 12 个月窗口

### 二、OpenAI Secure MCP Tunnel

#### 2.1 解决的核心问题

企业想让 AI Agent 访问内部数据（数据库、内部 API），但安全团队**绝不允许**内部服务暴露到公网。

#### 2.2 架构：纯出站连接

```
企业内网                           OpenAI 控制平面
┌──────────────────┐              ┌─────────────────┐
│  tunnel-client   │──HTTPS出站──→│  请求队列         │
│  (sidecar容器)   │  长轮询      │  api.openai.com  │
│       │          │              │       │          │
│       ↓          │              │       ↓          │
│  本地MCP Server  │              │  ChatGPT/Codex   │
│  localhost:8080  │              │  Responses API   │
└──────────────────┘              └─────────────────┘

   零入站端口     MCP Server地址    认证密钥
   防火墙     永不暴露到公网     不出企业边界
```

**安全保证**：
- 企业防火墙上零入站端口开放
- MCP Server 地址从不暴露到公网
- 认证密钥不离开企业边界
- 支持 mTLS、出站代理、自定义 CA

#### 2.3 部署方式

**CLI 快速启动**：
```bash
export CONTROL_PLANE_TUNNEL_ID="tnl_dev_your_unique_id"
export CONTROL_PLANE_API_KEY="your_runtime_key"

# 指向本地进程
tunnel-client run --command "uv run local-mcp-server"

# 指向已有内部 HTTP 端点
tunnel-client run --mcp-server-url "https://mcp.internal.example.com/mcp"
```

**Kubernetes Sidecar（生产环境推荐）**：
将 tunnel-client 作为 sidecar 容器部署在与 MCP Server 同一个 Pod 中，连接 `http://localhost:8080/mcp`，通过出站 HTTPS 安全连接 OpenAI 控制平面。

#### 2.4 运维工具

- 健康检查：`/healthz`、`/readyz`
- 可观测性：`/metrics`（Prometheus 格式）
- 管理 UI：`/ui`
- 调试：`tunnel-client doctor --profile --explain`

### 三、对 AI Agent 开发的影响

#### 3.1 架构重构启示

| 变化 | 影响 |
|------|------|
| MCP 无状态化 | Agent 可对接高可用、自动扩缩的后端服务，不再是单实例持久连接 |
| 安全隧道 | 企业 AI Agent 最大的拦路虎——访问内部数据——被消除，无需 VPN 或公网端点 |
| MCP Apps | 开发者可为 Agent 构建可视化交互界面，用户不再被迫在聊天框输入复杂 JSON |
| Tasks Extension | 模型训练、大规模数据处理等长任务可被 Agent 触发并异步跟踪 |

#### 3.2 开发者行动计划

1. **重构有状态逻辑**：审查现有 MCP Server，移除对 `initialize` 握手和 Session ID 的依赖，必要上下文放入 `_meta` 字段
2. **实现缓存**：利用 `ttlMs` 和 `cacheScope`（SEP-2549）缓存工具定义，减少延迟和上下文窗口膨胀
3. **准备长任务**：将阻塞操作重构为返回 Task 句柄，在客户端实现 `tasks/get` 轮询
4. **采用隧道模式**：企业场景立即集成 tunnel-client sidecar 到 Docker/K8s 部署
5. **学习扩展机制**：后续功能将通过扩展交付，理解 `extensions` 协商机制以便渐进采纳

#### 3.3 2026 路线图前瞻

- **Triggers**：Webhook 能力，服务器可主动通知 Agent 新数据到来
- **Native Streaming**：工具结果的增量返回（对 LLM Token 流式输出至关重要）
- **Skills**：服务端打包的领域知识，指导 Agent 如何最优使用特定工具
- **Progressive Discovery**：按需加载工具，而非将所有工具定义注入上下文窗口
