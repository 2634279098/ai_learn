# 从跟风编程到 Agent 工程 — Claude Code 12 大最佳实践

## 一句话总结
基于 Claude Code 缔造者 Boris Cherny 审阅的 69 条建议，提炼出从"跟风编程（Vibe Coding）"到"Agent 工程（Agent Engineering）"的范式转变——不是换工具，而是以不同方式使用同一工具。

## 适用场景
- 已使用 Claude Code 一段时间但停留在"描述→尝试→纠正→重复"循环的开发者
- 希望将 AI 编程能力从个人效率提升扩展到团队工程化管理的技术负责人
- 正在探索如何通过配置文件（而非每次手写提示词）来系统化管理 AI 编程

## 详细内容

### 跟风编程 vs Agent 工程

| 维度 | 跟风编程 | Agent 工程 |
|------|---------|-----------|
| 约束来源 | 每次手写提示词 | 配置文件 + 权限限制 |
| 工作方式 | 描述→尝试→纠正→重复 | 预设规则→自动执行→自动验证 |
| 上下文管理 | 手动 `/compact` | `context: fork` 自动清理 |
| 权限控制 | 每次手动批准 | Agent 物理上无权执行 |
| 可复用性 | 每次重写提示词 | Skills/Commands/Agents |
| 并行度 | 一次一个对话 | 几十个 Agent 同时运行 |
| 验证方式 | 肉眼检查 | Stop 钩子自动验证 |

### 12 大最佳实践

#### 1. 优化 CLAUDE.md，避免规则冗余

- **60 行最优，200 行是上限**
- 过长文件会导致靠后的规则被降权忽略
- 使用 `<important if="language=go">` 等条件标签，让规则仅在相关语境下激活
- 采用多层目录结构（根目录通用规则 + 子目录领域规则）

#### 2. 将 Claude Code 视为编排系统，而非聊天窗口

停止每次重新输入完整提示词。利用 `.claude/` 目录下的三种配置：

| 配置 | 作用 | 示例 |
|------|------|------|
| `commands/` | 可复用工作流快捷方式 | `/changelog`、`/review` |
| `agents/` | 受限工具访问的专用子 Agent | 创建仅有读取权限的 `reviewer.md` Agent |
| `skills/` | 跨项目知识模块 | `new-microservice` 技能包 |

#### 3. 将上下文视作资源进行管理

**关键洞察**：Skills 默认共享主 Agent 上下文，会导致响应变慢。

**解决方案**：使用 `context: fork` 让 Skill 在隔离子 Agent 中运行，完成后丢弃上下文，主 Agent 只接收摘要输出。

#### 4. 善用 Hooks（钩子）实现旁路自动化

三个钩子在 Agent 主循环外运行，**不消耗 token**：

| 钩子 | 用途 |
|------|------|
| **PreToolUse** | 写入前自动格式化（gofmt）、防护破坏性操作 |
| **PostToolUse** | 写入后自动验证、日志记录 |
| **Stop** | 完成验证（测试通过？文件存在？API 响应正常？） |

#### 5. 使用沙盒模式减少权限弹窗

- `/sandbox` 在隔离环境中运行 shell 命令，限制爆炸半径
- **效果**：减少 **84% 的权限弹窗**

#### 6. 并行运行多个 Claude 实例

- 使用 `claude -w` 在 git worktree 内启动隔离实例，结合 tmux 多面板
- **效果**：多个 Agent 同时处理不同任务（重构、数据库、测试），互不干扰

#### 7. 利用 ultrathink 触发深度推理

- 在提示词中加入 `ultrathink` 关键词触发更高努力推理模式
- 能浮现常规模式遗漏的约束和问题，适用于关键架构决策

#### 8. 使用 `/loop` 实现本地开发自动化

- `/loop 30m /code-review` 可实现定时自动化工作流
- 与 CI 不同：能访问当前工作状态和历史上下文，在开发过程中而非合并后发现问题

#### 9. 用 `/btw` 提问不中断主任务

- `/btw <问题>` 将旁路问题加入队列，Claude 在自然暂停点处理后再回到原任务
- **价值**：避免中断长流程任务导致上下文丢失

#### 10. 以 Gotchas（踩坑记录）为 Skill 核心

- Skill 文档中最有价值的部分是实际失败的记录，而非概述或示例
- **原则**：先写 Gotchas，每次 Claude 失败就更新，将失败转化为未来预防

#### 11. 用 Stop 钩子验证任务真正完成

- Claude 声称完成不代表真正完成
- **方案**：Stop 钩子连接验证脚本（测试、文件检查、API 验证），失败则回滚任务状态

#### 12. 使用 `--bare` 参数加速 SDK 调用

- `claude --bare --prompt "..."` 跳过上下文发现过程
- **效果**：启动时间减少最高 **10 倍**，批量自动化场景尤为显著

### 核心架构思维

#### 1. 代码审查 Agent 示例

创建一个仅有**读取权限**的 `reviewer.md` Agent，从配置层面禁止修改文件：

```markdown
---
name: reviewer
tools: Read, Grep, Glob
---
你是一个代码审查专家。你的任务是审查代码质量、发现潜在问题、
提出改进建议，但你绝对不能修改任何文件。所有发现以报告形式输出。
```

#### 2. context: fork 隔离执行

```markdown
---
name: complex-refactor
context: fork
---
执行大型重构任务。fork 模式确保重构过程不会污染主 Agent 上下文，
完成后只将最终结果摘要返回主 Agent。
```

#### 3. 从"跟风"到"工程"的转变清单

- [ ] 用配置文件替代每次手写提示词
- [ ] 用 Agents 的权限限制替代手动批准
- [ ] 用 `context: fork` 替代手动 `/compact`
- [ ] 用 Skills/Commands 替代重复提示词
- [ ] 用并行 Agent 替代单任务串行
- [ ] 用 Stop 钩子自动验证替代肉眼检查

---

**核心结论**：以上所有功能（agents、skills、hooks、worktrees、loops）都是 Claude Code 自带的内置能力。转变不在于购买新工具，而在于重新配置和组合你已有的工具。
