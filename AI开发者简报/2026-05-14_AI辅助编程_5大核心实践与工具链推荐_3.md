## 一、第一原则：先思考，再让 AI 写

**核心理念**：不要把思考外包给模型，AI 是工具而非大脑。

**正确工作流程**：
1. **自己先想清楚**：手写伪代码、画流程图、列关键数据结构
2. **完整输入上下文**：将你的思路 + 约束 + 背景完整提供给 AI
3. **让 AI 负责执行层**：实现细节、填充 boilerplate、优化性能、补测试用例

> 💡 **角色定位**：你 = 架构师 + Code Review 组长 + 测试负责人；AI = 超级靠谱的实习生

## 二、Prompt 四个黄金要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **① 角色定位** | 具体化身份、年限、风格 | "12年+ Go后端专家 + 前字节架构师 + 极致性能&可读性追求者" |
| **② 完整上下文与硬约束** | 技术栈、分层、规范、性能 SLA、安全红线 | 性能红线：P99 < 80ms，QPS > 2000 |
| **③ 强制结构化输出指令** | 编号 + "严格按顺序输出，不要打乱" | 锁死输出步骤，防止跳步或直接甩代码 |
| **④ 验收标准/死亡红线** | 明确不可接受的底线 | 防缓存雪崩、错误规范化、禁止循环内阻塞 IO |

### Prompt 模板结构

```
角色：[具体身份 + 年限 + 风格倾向]
上下文：[技术栈 + 架构分层 + 已有规范 + 性能/安全约束]
任务：[具体需求 + 入参出参定义]
输出指令：[严格编号的步骤，不可打乱]
验收标准：[死亡红线清单]
```

## 三、任务粒度铁律：小步快跑

**为什么不能一次做太大？** 大需求容易导致：
- 前后假设不一致
- 细节偷懒 / 产生幻觉
- 引入"看起来能跑但全是坑"的代码
- 输出巨型 diff，review 直接崩溃

**最有效节奏**：
- ✅ 一次只完成 **1-2 个核心函数** 或 **一个小的垂直切片**
- ✅ 编译通过 → 测试通过 → 风格 OK → 再继续下一块
- ✅ 每轮 Prompt 带上**上一次输出 + 具体反馈**
- ✅ 用"增量式生成 + 增量式重构"代替"大爆炸式一次性生成"

> 🔑 **铁律**：宁可写十个小 Prompt 各产生高质量的 150-300 行代码，也不要写一个大 Prompt 产生 5000 行需要全部重写的垃圾。

## 四、必须建立的防护栏

| 防护措施 | 具体操作 |
|----------|----------|
| **代码审查** | 编译 → 单元测试 → golangci-lint/staticcheck → 人工 diff，**永远不直接合入未审查代码** |
| **安全双确认** | 安全、鉴权、加密相关代码必须双人确认 |
| **大范围改动预检** | 用 diff 工具先整体看一遍（Agent 经常一次改 30+ 文件） |
| **保持手写习惯** | 每周至少 1-2 天不用 AI，写最核心的业务逻辑，保持手感和判断力 |

## 六、AI 辅助编程的三个真实阶段

```
阶段一【工具人】（大多数人）
  └─ 复制粘贴 → 修 bug → 问语法

阶段二【提效阶段】（中上水平）
  └─ 主动写好 Prompt → 控制粒度 → 快速迭代

阶段三【系统性红利】（顶级 10%）
  └─ 建立 Spec → 代码 → 测试 → 文档 闭环
  └─ 一个人干 3-5 个人的活，代码质量更稳定
```

## 📌 核心总结

> **2026 年的竞争核心**不再是"会不会写代码"，而是**"会不会用 AI 大规模、高质量地写代码"**。

### 关键能力三要素

1. **思考在前** —— 先想清楚架构和边界
2. **Prompt 精准** —— 上下文完整、约束明确
3. **小步迭代** —— 拆分任务、增量推进

### 防护四要点

1. 代码必须 Review，不能直接合入
2. 安全相关代码双人确认
3. 大范围改动先整体预览
4. 保持手写习惯，不完全依赖 AI
