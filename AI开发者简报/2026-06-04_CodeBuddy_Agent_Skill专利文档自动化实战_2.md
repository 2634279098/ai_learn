# CodeBuddy Agent + Skill：AI 驱动的专利文档自动化实战

## 一句话总结
鹅厂开发者使用 CodeBuddy + patent-writing Skill，将专利技术交底书从「改三遍才能提交」变为「2 小时 AI 自动生成、代理人说改一遍就能提交」，核心架构是「AI 负责理解和写作 + 代码引擎负责输出规范文件」的三层分离设计。

## 适用场景
- 需要批量生成结构化文档（专利、技术方案、项目报告）
- 搭建 AI Agent + Skill 自动化工作流
- 学习「声明式 JSON + 构建引擎」的文档生成架构模式
- 非开发岗位（产品、设计、法务）利用 AI Agent 产出专业文档

## 详细内容

### 一、问题：专利文档的痛点

提交发明专利交底书需要覆盖以下内容：
- 关键术语定义
- 发明构思（三段式）
- 背景技术 + 现有技术缺点
- 发明内容：产品侧 + 技术侧（含完整步骤）+ 有益效果
- Mermaid 流程图
- 参考文献

**之前的痛点**：
- 光看格式要求就花半天，还是云里雾里
- 技术人员有方案，但「翻译」成专利语言耗时巨大
- 提交后被代理人打回重写，反复 3 遍以上

### 二、方案：三层分离架构

```
┌─────────────────────────────────────────────────┐
│  Layer 1: 理解层（AI）                            │
│  读取代码/文档/图片 → 提取技术方案 → 识别创新点      │
├─────────────────────────────────────────────────┤
│  Layer 2: 写作层（AI）                            │
│  按专利规范重写 → 专利语言（非工程术语）             │
│  输出结构化 JSON（patent_content.json）           │
├─────────────────────────────────────────────────┤
│  Layer 3: 文件生成层（代码引擎）                    │
│  JSON → Word 模板引擎 → 标准 .docx 文件            │
│  Mermaid → mmdc 渲染 → PNG 自动插入               │
└─────────────────────────────────────────────────┘
```

**关键设计原则**：AI 永远不直接操作 Word XML。它只写声明式 JSON（「我要什么」），代码引擎执行。这样：
- AI 的「幻觉」不会破坏文件结构
- JSON 可独立调试和增量修改
- 模板可独立升级，不影响内容生成逻辑

### 三、完整 7 阶段工作流

```
① 信息采集 → ② 技术分析 → ③ 大纲确认(⛔人工唯一卡点)
→ ④ 内容写作 → ⑤ 流程图渲染 → ⑥ DOCX 组装 → ⑦ 验证
```

| 阶段 | 执行者 | 说明 |
|------|--------|------|
| ① 信息采集 | AI 自动 | 自动检测输入模式（代码/文档/图片/文字），确认发明名称和保护目的 |
| ② 技术分析 | AI 自动 | 深度分析系统架构、算法、创新点、技术细节 |
| ③ 大纲确认 | ⛔ **人工** | AI 给出逐章大纲和关键点，**用户确认后才开始写作** |
| ④ 内容写作 | AI 自动 | 按专利语言规范逐章写作，公式用 OMML 格式标注 |
| ⑤ 流程图渲染 | 引擎自动 | Mermaid 语法 → mmdc 渲染 → 高清 PNG |
| ⑥ DOCX 组装 | 引擎自动 | 读取 JSON → 填充 Word 模板 → 输出标准 .docx |
| ⑦ 验证 | 引擎自动 | 验证文件可正常打开，报告输出路径 |

**总耗时**：约 2 小时（大部分是 AI 处理时间，用户实际投入约 5-10 分钟审核大纲）

### 四、为什么卡点设在大纲阶段？

作者的核心理念：

> 「在文字层面改逻辑，5 分钟搞定；等到 Word 文件生成之后再改结构，至少 30 分钟。卡点设在最前面，代价最小。」

这是整个工作流中**唯一的人工检查点**，设计思路是：
- 前期投入最少时间可获得最大修改杠杆
- 一旦大纲确认，后续全部自动化
- AI 不会在错误的框架上浪费 Token

### 五、输入灵活性：什么都能扔进去

支持三种输入模式同时混用：

```
模式 A1: 代码文件/文件夹  →  分析架构和算法
模式 A2: 设计文档 PPT      →  提取产品方案
模式 A3: 架构图/截图      →  图像识别转换
```

作者实际操作：「同时扔了代码 + 架构图 + 一份设计说明 PPT」

### 六、JSON 内容结构：6 种段落类型

`patent_content.json` 使用声明式类型描述所有专利内容：

| 类型 | 用途 | 格式 |
|------|------|------|
| `text` | 正文段落（步骤描述、解释说明） | 楷体 10pt |
| `heading` | 章节标题 | 微软雅黑 Bold 10pt |
| `formula` | 独立公式 | 居中，右侧编号(N) |
| `inline` | 行内数学公式 | 混排在文字中 |
| `image` | 图片及图注 | 居中，可配置宽度 |
| `empty` | 间距控制 | 空行 |

**关键细节**：数学公式使用 **Office 原生 OMML 格式**，不是 LaTeX。这意味着专利代理人在 Word 中可直接编辑公式——无需插件。

### 七、增量修改：不必从头开始

所有中间产物都被保留：

```
patent_content.json     → 编辑特定章节
mermaid_source.mmd      → 修改流程图
template_directory/     → 调整模板如有需要
```

代理人要求修改时的工作流：
1. 打开 `patent_content.json`
2. 找到对应章节（如 4.2）
3. 改那几行文字
4. 重新运行构建脚本
5. **约 3 分钟输出新 .docx**

### 八、最终交付物

生成的 `.docx` 包含：

- ✅ 头部信息表格（发明名称、涉及产品、保护目的）
- ✅ 关键术语定义（7 个核心概念）
- ✅ 发明构思（3 段，语言风格已符合规范）
- ✅ 背景技术 + 现有技术缺点
- ✅ 发明内容：产品侧 + 技术侧（S1-S8 完整步骤）+ 有益效果
- ✅ Mermaid 流程图渲染为 PNG 自动插入
- ✅ 参考文献

**质量验证（专利代理人评价）**：
> 「框架完整，语言基本合格，改一遍就能提交」
> 
> 对比之前：「以前我写的初稿要改三遍」

### 九、可复用的架构启示

1. **声明式 JSON 是 AI 和代码引擎之间的「合同」**——AI 输出想法，代码引擎执行，互不越界
2. **单卡点策略**——整个人工干预压缩到一个节点（大纲确认），最大化自动化覆盖率
3. **保留中间产物**——增量修改成本从「小时级」降为「分钟级」
4. **OMML 而非 LaTeX**——面向真实用户（用原生 Word 的专利代理人），而非技术用户
5. **安装即用**——在 CodeBuddy 中直接安装 `patent-writing` Skill，无需额外配置
